# BLE セントラル × 供給 マトリクス（実機・2026-07-17）

**DUT**：C3 `<MAC-19>`（`1-6:1`）／C5#1 `<MAC-37>`（`1-6:3/4`）／
C6 `<MAC-03>`（`1-6:2`）。**central**＝ユーザーの iPhone / Android 実機、および BlueZ `hci0`。
**測定者**＝親エージェント（LP_AON 直読み）。**スマホ操作＝ユーザー。**

## 0. ★このラウンドの最大の成果は «自分の測定手順が偽の現象を作っていた» ことの発見

**本日、親は「Android で connect できない」を C3 で3回・C5 で2回「再現」した。それは交絡だった。**

- **デバイスの bond ストアは RAM backed**（同日 C5 で3通りに確定：`ble_store_nvs.c` 非リンク／
  `PERSIST=0`（`esp_nimble_cfg.h:1097` が未定義マクロ依存）／ELF に `nvs_open` 0個）。
- ⇒ **親が «セルをクリーンにする» ために行った電源断が、毎回「スマホは鍵を持つ／デバイスは失う」
  という不一致を作っていた。** `ETIMEOUT`(13) も `ENOTCONN`(7) も「鍵が合わない」の症状として辻褄が合う。
- ⇒ **さらに悪いことに、鍵を持ったスマホが «自動再接続» してボードを掴み、次のセルまで汚した**
  （`CONN` は毎起動クリアなので、詰まった接続が «電源投入の後» に起きたことが物証）。
- **ユーザーが両スマホで forget した測定だけがフレッシュで、それは通った。**

★**判別子（確立）**：**`PAIR` タグ（`0x5DC0`）が立つ＝その接続でペアリングが実行された＝フレッシュ。
立たない＝既存 bond の再利用を試みた＝交絡の疑い。**
★**運用**：**全セルで «両方の» スマホを forget → BT OFF/ON**（iPhone のセルでも Android を forget）。
これは**キャッシュ衛生ではなく鍵不一致の予防**。

> サブエージェントは最初から「**auto-reconnect is the matrix's hidden enemy — forget on all phones
> before every cell**」と警告していた。親はそれを «手順の注意» として扱い、**«測定を根本から壊す交絡»
> だと理解していなかった**。機序（RAM backed）を突き止めるまで9時間、存在しないバグを追わせた
> （PVCY draft の実機投入まで行った）。

## 2. 読み取れること（★因果は主張しない）

1. **ユーザー仮説「供給を替えると通るセントラルが入れ替わる」は 6/8 で当たり、2つ外れる**：
   - **C3 idf × iPhone**：仮説 OK → 実測 **NG**
   - **C6 idf × Android**：仮説 NG → 実測 **OK**
   ⇒ **供給だけでは説明できない。チップも効いている。**
2. **チップ別に見ると単純**：**C6 だけが両方通る／C3 は iPhone で必ず落ちる（供給不問）／C5 は Android で必ず落ちる**。
3. ★**「NG」の中身が2種類ある**：
   - **«デバイスは成功したのに相手が待ちきれない»**（C3 × iPhone・両供給）＝`PAIR`/`ENC` とも status 0。
     **デバイス側に異常が無い。** ユーザー側も「デバイスとして登録されている」＝**双方が bond 成立と認識**。
   - **«暗号化に到達しない»**（C3 idf × Android＝`ETIMEOUT`／C5 idf × Android＝`ENOTCONN`）。
4. ★**「詰まる」かどうかがチップで分かれる**：
   - **C5/C6**：iPhone が timeout しても **切断（reason `0x13`）が届く** ⇒ **再広告 → 再接続できる**
     （ユーザー報告＝「1回目の disconnect でエラーが出る以外は OK。以後は何回でもクリーン」）。
   - **C3**：**切断が届かない**（`DISC=0`）⇒ **リンクを握ったまま広告停止 ⇒ 復旧はリセットのみ
     ⇒ そのリセットが RAM の bond を消す**＝**回復不能のループ**。
   ⇒ **C3 の実害は «timeout» ではなく «切断が来ずに詰まること»。**
5. **`our_sec=1` はこのプロジェクト初の観測**（C3 hal × Android）。同日のレビューが
   「`our_sec>=1` は満たされた実績が皆無（全文検索で `our=<非0>` 0件）＝ソースコメント由来の期待値」
   と報告していた基準が、**初めて満たされた**。

## 3. 未確定・申し送り

- **C3 × iPhone が «本当に NG か» は未確定**。**双方 bond 成立を認識**しており、C5/C6 の
  「最初の切断だけエラー表示」と同型の可能性がある。**確かめるには «リセットせずに再接続»
  が要るが、C3 は詰まるのでリセット無しには広告に戻れない ⇒ 現行の計装では原理的に測れない。**
- **C6 が «なぜ» 両方通るのかは未測定**。C5/C6 は BT blob が **submodule ≡ hal（バイト同一）**、
  C3 は3方向すべて別物——だが**それは C5 と C6 の差を説明しない**。
- **iPhone の «最初の切断エラー» は C5/C6 でデバイス側 reason `0x13`＝正常**＝**直すべきものが
  見当たらない**（ユーザー判断＝「以前は再コネクトできなかったので進んでいる。このままで良い」）。
- **PVCY draft(ii)（`ESP32C3_BT_PVCY_FILTER`）は BlueZ を壊す**（1/4 vs baseline 7/7・
  **同一セッションで baseline を測り直した対照**で確定）＝出荷不可。**draft(i) は未着手・別機序。**
  ★**この結果は RAM-backed 交絡では説明できない**（両アームに等しく効くため）。
- ★**Android/RPA 対応は «要件»**（ユーザー明言）。**コード整理後に再検討**。

## 4. 測定の作法（このラウンドで確立・次ラウンドは必ず守る）

1. ★**全セルで両スマホを forget → BT OFF/ON**（鍵不一致の予防。キャッシュ衛生ではない）。
2. ★**1ビルドにつき1端末**。**`ENC` マーカは «最後の1件» しか保持しない**——iPhone が後で成功すると
   Android の失敗が**上書きされて消える**（親が一度見失った）。
3. ★**セルごとに «全マーカーを明示クリア» してから真cold**。**`ENC`/`PAIR` は毎起動クリアされない**
   ＝クリアしないと**前セルの成功値が «今回も成功» と読める**（C6 セル2 で実際に `0x5de00000` が残っていた）。
4. ★**マーカー表はアドレスでなく «タグ» で引く**。**全 STORE が使用中で共用が常態**：
   C3 `0x58`＝WRITE/BOOT_TRACE/**ENC**（last-wins）／C5 `0x600B1018`＝**ENC と PAIRING の共用**
   （★C5 は **PAIRING が ENC の «後» に発火**＝タグが `0x5DE0` のままなら PAIRING 未到達）／
   C6 `0x600B1018`＝ENC と reset_cb(`0x5E00`) 共用。
5. ★**読む（`--after no-reset`）と DUT が download mode に落ちて広告が止まる**。**ユーザーが試す前に読むな。**
   **`--after hard-reset` で戻すと «その起動» が GAP マーカを消す**（C6 は STORE6 のみ残存）
   ＝**GAP マーカは «起動させる前に» 読む**。
6. ★**hard-reset も電源断も RAM の bond を消す**＝**復旧操作そのものが次の測定条件を変える。**
