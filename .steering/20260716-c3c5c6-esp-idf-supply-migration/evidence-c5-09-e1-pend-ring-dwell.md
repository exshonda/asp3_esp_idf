## 4. ★E1 の設計（★計器は «ミラーのみ»）

### 4.1 何を測るか

| 量 | 意味 | ①にとっての役割 |
|---|---|---|
| ★**dwell high-water**＝`shim_que_pend_total` を **独立タスクから 200ms 周期でサンプルした最大値** | ★**「保留リングに «居座った» 個数」**＝**滞留そのもの** | ★**主判定** |
| ★**`shim_que_pend_used`**＝**保留経路の累積利用回数**（既存カウンタ） | **「E_CTX フォールバック経路が «そもそも使われたか»」** | ★★**較正**（下記） |

### 4.2 ★★なぜ `pend_used` を «必ず» 同時に測るか（＝0 を «盲» と区別するため）

★**dwell high-water == 0 は、2 つの «全く違う» 状態と両立する**：
1. **保留経路は使われたが、常に即座に flush された**（＝**滞留は無い**＝★**①が死ぬ**）
2. **保留経路が一度も使われていない**（＝**測定対象が存在しない**＝★**①について何も言えない**
   ＝**レビュー §5-b「0 を読んだら測定対象の存在を確かめる」の型**）

⇒ ★**`pend_used` がこの 2 つを分ける。** ★**`used>0 かつ hw==0` で初めて「滞留は実在しない」と言える。**
⇒ ★**`used==0` なら、E1 は «①を殺す» のではなく «①の前提が土俵に乗っていない» を意味する**
（**その場合、①は «C5 の Android 失敗» の説明として «機構ごと不在»＝別の意味で死ぬ**）。

### 4.3 ★なぜ «サンプル» で測るか（＝push 点の high-water では «滞留» を測れない）

★**push 点で `pend_total` の最大値を取っても、それは «滞留» ではない**：
**push 直後の `exit_critical` で即 flush されれば `0→1→0` となり、push 点 high-water は 1 を記録するが
«滞留時間はマイクロ秒»。** ⇒ ★**push 点 high-water は «経路が使われたか»（＝`used`）と実質同義。**
⇒ ★**滞留＝«居座り» を測るには «独立した時計から覗く» しかない**＝**200ms サンプラ**。
★**①が主張する滞留は «30 秒スケール»**（SM が対向 PDU を待つ）⇒ **200ms サンプラなら ~150 回捕捉できる**
＝**計器と仮説のスケールが整合している。**

★**正直な限界**：**200ms «未満» の滞留は取りこぼす。** だが★**そのような滞留は①が主張する病態
（対向がペアリング中に諦める＝秒スケール）を作れない**⇒ **仮説に対しては十分な感度。**

### 4.4 計器の実体（★最小・hot path に置かない）

- **shim 側**：`static` を app から読むための **アクセサ 1 個のみ**を追加（`TOPPERS_C5_PEND_DIAG` 既定 OFF）。
  ★**push/flush の hot path には «1 命令も» 足さない。**
- **app 側**：**既存の `storm_monitor_task`（200ms 周期・`MONITOR_PRIORITY=15`）**でサンプルし、
  **LP_AON へ 1 レジスタ書込み**（既存の STORE4 書込みを置換＝**書込み回数は増えない**）。
- ★**`--wrap` は «使わない»**（噛んだかの確認が要る計装を避ける＝レビューの規則）。

### 5.2 ★P_E1 を 15% にした理由（★レビュー 45%／親「35%より高いはず」の «どちらよりも低い»）

★**下げた理由（★§3 の一次情報が主）**：
1. ★★**pend_ring 滞留説は C3 で «直接介入»（100ms flush＝E2 そのもの）により反証済み**
   ——**レビューの «機構は実測確定» は出典の誤読**（§3.2）。
2. ★★**«flush 有» の 3 セルは «flush が効いた» 証拠を 1 つも持たない**
   （**C3=効かなかったと実測／C6=予防的**）＝**親の補正表の «整合» は中身が空**（§3.4）。
3. ★**機会的 flush が死んでいない**：`exit_critical` の最外解除ごとに走り、
   **BT クリティカルセクションは接続イベントごとに出入りする**＝**drain 機会はほぼ連続**（§2.3）。
4. ★**C5 の病態は «暗号前»（ENOTCONN）**＝**C3 の «暗号後の鍵配布待ち»（30 秒）とは段が違う**。
   ①の物語（SMP PDU が滞留 → SM が待つ）は **«暗号後の鍵配布» を想定した機構**であり、
   **C5 の «暗号に到達しない» を素直には説明しない**。
5. ★**`memory` の «C5×iPhone は flush 無しで OK»**＝**flush 欠如は単独で非対称を説明しない**
   （レビュー自身の正直な自己評価 §9）。

★**0 にしない理由（＝①を «測る前に» 殺さない）**：
1. ★**C3 の反証実験は «交絡» していた可能性**：当時 `PVCY=0` という **別の支配的真因**が同時に存在した
   ⇒ **「flush を足しても直らなかった」は «PVCY=0 のせい» でも説明がつく**
   ＝★**反証条件そのものを疑う**（§6.2 で独立に検算する）。
2. ★**C5 は 4 セル中で «唯一» 周期 flush を持たない**——**構造上の非対称は実在する**（§1.2 で実測）。
3. ★**原因が «我々の統合» に確定した**（`evidence-stock-01 §20`＝**読むだけ**）
   ⇒ **①は «我々の app»＝射程内**。
