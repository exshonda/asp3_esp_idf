## 2. ★マーカー全0の確認（依頼 (a)）— **罠は «実在» した**

### 2.1 クリア前＝**セル1 の `ENC_CHANGE` が生き残っていた**

```
0x600B1000 = 0x5ade51c0   0x600B1008 = 0x0ade5000   0x600B100C = 0xad000000
0x600B1014 = 0x000016ad   ★0x600B1018 = 0x5de00000  0x600B101C = 0xa1020704
0x600B1020 = 0x00000000   0x600B1024 = 0x00000000
```

★**`STORE6(0x600B1018) = 0x5de00000` ＝ セル1（iPhone）の暗号化成功マーカがそのまま残っていた**
⇒ **クリアしなければ «Android も暗号化に成功した» と誤読される**（＝依頼が警告した罠が**実在**）。
★**`STORE8/9` は既に 0** だった＝**私が `evidence-c6-07 §5.3` で実測した「STORE8/9 は毎起動 0 クリア
（`:757`/`:758`）／STORE6 だけがクリアされない」と完全に一致**（コーディネータの `--after hard-reset`
がそれを実行していた）。**「残るのは STORE6 だけ」という私の記録が、そのまま今回の危険箇所だった。**

### 2.2 クリア後＝**8本すべて `0x00000000`**（読み戻しで確認）

`0x600B1000`／`1008`／`100C`／`1014`／`1018`／`101C`／`1020`／`1024` ＝ **全て `0x00000000`**。

★**`STORE1(0x600B1004)` と `STORE4(0x600B1010)` は «触っていない»**：
**C6 ではそれぞれ `RTC_SLOW_CLK_CAL_REG`／`RTC_XTAL_FREQ_REG`＝実在するハードウェア・レジスタ**
（`evidence-c6-04` の申し送り）。**POR が処理する**ので明示クリアは不要かつ危険。

### 2.3 真cold ブート後＝**`STORE6` は `0` のまま**（★これがセル2 の前提）

```
STORE0 sync       = 0x5ade51c0   ← NimBLE sync
STORE2 adv        = 0x0ade5000   ← adv 試行
STORE3 adv rc     = 0xad000000   ← ble_gap_adv_start rc=0
★STORE6 ENC      = 0x00000000   ← ★未ペアリング＝セル1 と混ざらない
STORE7 intr       = 0xa1020704   ← storm 無
STORE8 CONNECT    = 0x00000000
STORE9 DISCONNECT = 0x00000000
```
真cold の証明＝`uhubctl -l 1-6 -p 2 -a off` ＋ **by-id 読み戻し 0**。
★**最後の `--after hard-reset`（広告開始）も `STORE6` を書かない**ので **0 のまま**。

## 4. ★ユーザー向け手順（セル2＝Android・依頼 (d)）

### 4.1 ★★**先に «両方» のスマホで forget**（＝今日の交絡の正体）

> **デバイスの bond ストアは RAM backed**（`ble_store_ram`／`ble_store_config` PERSIST=0）
> ⇒ **電源断のたびにデバイスは鍵を失うが、スマホは鍵を持ったまま**＝**不一致**。
> これが `ETIMEOUT(13)`／`ENOTCONN(7)` の «Android は失敗する» という観測群を作っていた。

1. **★Android**：`ASP3-C6-BLE` を **forget（登録解除）** → **Bluetooth を OFF → ON**
2. **★iPhone も `ASP3-C6-BLE` を forget**
   — ★**iPhone は «今» bond を持っている**（セル1 で成立した）。
   **放置すると自動再接続してセルを汚す／ボードを掴んでしまう**。
3. **この1セルは Android «だけ»**。**iPhone を挟まないでください**
   （`ENC_CHANGE` は **最後の1件しか残らない**＝上書きされると判定不能）。

### 4.2 手順（Android・nRF Connect 等）

| # | 操作 | 期待 |
|---|---|---|
| 0 | スキャン → **`ASP3-C6-BLE`** に CONNECT | 接続 |
| 1 | サービス **`0xABF0`** を開く | `0xABF1`〜`0xABF4` の4特性（見えなければ **4.1 のキャッシュ**を疑う） |
| 2 | **`0xABF4` を READ** | **未ペアなら弾かれてペアリング要求 → Just Works で bond → 再READで `"BT4-OK"`** |

★`sm_io_cap=NO_IO`・`sm_bonding=1`・**`sm_sc=1`** ⇒ **PIN 入力は出ません**。
★**うまくいかない場合も «そのまま» にしてください**（失敗の証拠がマーカーに残ります）。

## 5. ★あなたが読むべき STORE とタグ（依頼 (e)）

★**読むと広告が止まります**。**読了後は `--after hard-reset`**（★その起動が `STORE8/9` を
クリアし `STORE3` を上書きします＝**GAP マーカは «起動させる前に» 読む**）。

| addr | 意味 | タグ／期待 |
|---|---|---|
| **`0x600B1018`** | **ENC_CHANGE**（★D-2d の本体） | **`0x5DE0_00<status>`**：`0x5de00000`＝**成功** ／ `0x5de00007`＝`ENOTCONN`(7) ／ `0x5de0000d`＝`ETIMEOUT`(13)。★**`0x00000000` のままなら «暗号化に一度も到達しなかった»**。★`ble_hs` reset_cb と共用（tag **`0x5E00`**）＝**上位バイトで判別** |
| **`0x600B1020`** | GAP CONNECT | **`0x604E<status:8><count:8>`**（例 `0x604e0001`） |
| **`0x600B1024`** | GAP DISCONNECT | **`0xD15C<reason:8><count:8>`**（`0x13`＝正常切断） |
| `0x600B100C` | `0xABF3` write 受信 | `0x7717<count><先頭byte>`（★adv rc `0xad000000` と共用） |
| `0x600B1000` / `0x600B101C` | sync / intr | `0x5ade51c0` / `0xa1020704` |

★**`0x600B1010`(STORE4)＝`RTC_XTAL_FREQ_REG`・`0x600B1004`(STORE1)＝`RTC_SLOW_CLK_CAL_REG` は
判定に使わないでください**（app が診断ミラーに流用している潜在バグ＝未修正）。

★**フレッシュ判別子**：コーディネータの指摘どおり **`PAIR` タグが立てば «その接続でペアリング実行»＝
フレッシュ**／**立たなければ既存 bond 再利用＝交絡**。**本セルは `STORE6=0` から始まる**ので、
**非0 になれば «この接続でペアリングが走った»** ことになる。
