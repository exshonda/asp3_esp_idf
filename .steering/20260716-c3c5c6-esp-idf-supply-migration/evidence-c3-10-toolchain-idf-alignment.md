## 1. Phase 1 — `heap_caps_malloc` の根治

### 1.1 ★前提の訂正（親の引き継ぎ／evidence-c6-15 §1.6(a) の機序説明）

引き継ぎは「**hal 側のコピーが `heap_caps_malloc` を推移的に宣言する include を欠く**」
＝**hal 固有の問題**と述べていた。**実測すると両供給とも落ちる**：

| 経路 | 供給 | 箇所 | 結果（GCC14・手渡しフラグ無し） |
|---|---|---|---|
| C3 WiFi | **esp-idf**（`ASP3_ESPIDF_SUPPLY=ON`） | `esp-idf/components/esp_phy/src/phy_init.c:479` | **error: implicit declaration** |
| C3 BLE | **esp-idf**（`=ON`） | 同上 `:479` | **error** |
| C3 BLE | **hal**（`=OFF`） | `hal/components/esp_phy/src/phy_init.c:470` | **error** |

⇒ **「どちらの経路が生きるか」は `ASP3_ESPIDF_SUPPLY` で決まり，どちらでも落ちる。**

### 1.2 ★真の機序（＝shadow。hal 固有ではない）

`#include` を実測（読解ではなく両ファイルの実物）：

| 供給 | `esp_heap_caps.h` を include? | `freertos/FreeRTOS.h` を include? |
|---|---|---|
| **esp-idf** `phy_init.c` | **していない** | **している**（`:26`） |
| **hal** `phy_init.c` | **していない** | **していない**（代わりに `platform/os.h`） |

- **esp-idf 版**：実物の IDF ビルドなら `freertos/FreeRTOS.h` 経由で推移的に
  宣言が来る。**しかし本ビルドの `freertos/FreeRTOS.h` は
  `asp3/target/esp32c3_espidf/bt/stub/include/freertos/FreeRTOS.h`（82行の
  スタブ）に解決される**＝**推移経路が切れている**。
  ⇒ **原因は «我々のスタブが本物を shadow していること»**。
  ＝`esp_timer.h` の shadow（evidence-c6-15 §1）と**同一クラス**。
- **hal 版**：そもそも `FreeRTOS.h` を include しない世代＝**上流の素の欠落**。

★**インクルードパスは «既に» 通っている**（実測）：WiFi/BLE とも
`-I<supply>/components/heap/include` がコンパイル行に在る。
⇒ **「パスが無い」問題ではない＝宣言が読まれていないだけ**。

### 1.3 完了条件（★実測。ここが本 Phase のゴール）

### 1.4 シムの内容（★新規則を発明していない）

**`set_source_files_properties(<phy_init.c> PROPERTIES COMPILE_OPTIONS "-include;esp_heap_caps.h")`**

### 1.5 ★`-D MALLOC_CAP_*` の撤去（＝旧コメントの事実誤認の是正）

**実測**：`MALLOC_CAP_*` を使う**コンパイル対象**ファイルと、その include 状況：

| `bt/porting/mem/bt_osi_mem.c` | している | 冗長 |
| `mbedtls/port/esp_mem.c` | している | 冗長 |

⇒ `esp_bt.cmake` の旧コメント
**「`bt.c` は `esp_heap_caps.h` を #include せず直値のビットマスクを期待する」は偽**。
（`MALLOC_CAP_RETENTION` を使う esp-idf 版 `bt.c:1077` も同様に include 済み。）

★**-D は有害だった（実測）**：`-D MALLOC_CAP_DMA=8` と本物の `#define MALLOC_CAP_DMA (1<<3)` は

### 1.6 ★修正前後のコード生成差（計器を明記）

**md5 は不正な計器**（`__TIME__` 埋込み）なので使わない。
**計器＝オブジェクト単位の逆アセンブル比較**（再配置可能＝リンクアドレスのシフトを受けない）。
**objdump の `#` 注釈とローカルラベル名を正規化**（前エージェントが**同一入力で偽の差分**を
掴んだ既知のノイズ源。硬化しないと「自分の変更が別ファイルを変えた」と誤報告する）。

**比較は «同一コンパイラ（GCC15）» 上で行う**＝私の変更だけを分離する
（before＝HEAD＋手渡しフラグ／after＝本修正・フラグ無し）。

| 対照 | 結果 | 意味 |
|---|---|---|
| ★**ノイズ対照**（同一入力の2ビルド） | **179 SAME / 0 DIFFER** | **計器は偽陽性を出さない** |
| ★**較正（陽性対照）**：`bool` vs 暗黙 `int` | **DIFFER=1** | ★**計器は盲目でない** |
| **WiFi**：before vs after（GCC15） | **179 SAME / 0 DIFFER** | **no-op** |
| **BLE**（hal・NimBLE 実効 ON）：before vs after（GCC15） | **122 SAME / 0 DIFFER** | **no-op** |

★**「型互換だから無害」を前提にしていない — 実測して機序まで示した**：

```
較正（bool・evidence-c6-15 が報告した現象を私の環境で再現）：
  explicit bool : auipc t1 / jr t1                       ← 末尾呼出
  implicit int  : addi sp,-16 / sw ra / jalr / lw ra / snez a0,a0 / addi sp,16 / ret

本件（void* ＝ heap_caps_malloc のパターン）：
  explicit void*: lui a1 / addi a1 / li a0,84 / auipc t1 / jr t1
  implicit int  : lui a1 / addi a1 / li a0,84 / auipc t1 / jr t1   ← **完全一致**
```

**機序**：`bool` は戻り値を **{0,1} へ正規化**する必要があるので暗黙 `int` だと
呼出側に `snez` が挿入される。**`void *` と `int` は RV32 では
どちらも 32bit・同じ a0** で返り，`(uint32_t *)` へのキャストは
**純粋な再解釈＝命令を要さない**。⇒ **`bool` の前例は `void*` には当たらない**。
★**これは «前提» ではなく «測定の結論»**（較正済みの計器が差を検出できる状態で 0 だった）。

★**私の測定バグ（自己申告・2件）**：
1. **比較範囲が足りていなかった**：最初 `CMakeFiles/asp.dir` だけを見て
   「BLE 90 objects・DIFFER=0」と読みかけた。**実際の総数は 122**で、
   `CMakeFiles/asp3.dir`（30）と `cfg1_out.dir`（2）を**測っていなかった**。
   ⇒ **32 個を «測らずに 0» と報告する寸前**。`CMakeFiles` 直下へ広げて再測した。
2. **CMakeCache を «実効値» と読みかけた**：`ESP32C3_BT_NIMBLE:BOOL=OFF` と
   表示されるので「NimBLE が入っていない＝別構成を測っている」と誤断しかけた。
   **実効値は ON**（`option()` の cache を `set()` の通常変数が shadow する）。
   **実測で確定**＝nimble オブジェクト **63 個**・総数 **122**＝既知良好 `c3ble_halbk` と**一致**。
   ⇒ **cache の表示は実効値ではない。オブジェクトを数えて確かめた。**

## 4. ★★Phase 3 実測結果 — **bond は toolchain 依存だった**（予測 P3 は外れ＝重大な発見）

### 4.1 結論（先に）

**同一 DUT・同一 central（`hci0` `8C:1D:96:BA:6D:BD`）・同一セッション・毎回 forget・全て真cold**：

| アーム | 供給 | コンパイラ | **bond** |
|---|---|---|---|
| **A** | esp-idf（真 v5.5.4） | **esp-15.2.0（GCC15）**＝C3 の既知良好 | ★**失敗 0/5**（`AuthenticationCanceled`） |
| ★**B** | esp-idf（**同一**） | **esp-14.2.0_20260121（GCC14）**＝IDF 標準 | ★★**成功 5/6**（暗号必須 read=`BT4-OK`） |
| **C**（陽性対照） | **hal** | esp-15.2.0 | **成功**（＋既知良好 `c3_d2d` 2/2 成功） |

⇒ **供給・ソース・blob（`libbtdm_app.a`＝`859e8c8e`）は A/B で同一。違いはコンパイラだけ。**
⇒ ★★**「bond 失敗は供給に帰属」（evidence-c3-03 §5）は誤り。実体は «供給 × toolchain の交互作用»**
＝**esp-idf 供給は «IDF が指定する toolchain で建てれば» bond が通る。**

⇒ **ユーザーの「esp-idf で統一する」決定は bond を犠牲にしない**（guard とセットである限り）。
**toolchain guard はこの既定の «前提条件»**＝guard を切って GCC15 で建てるとアーム A になる。

### 4.3 ★帰属が成立する根拠（＝反証条件の充足）

事前登録：**「A 成功・B 失敗」でなければ toolchain に帰属してはならない／「A も B も落ちた」＝環境**。
本ラウンドは**その逆向き**（A 失敗・B 成功）だが、**同じ論理**で帰属する：

1. ★**陽性対照が同一セッションで成功**（hal 供給＝C）⇒ **central／ハーネス／電波の問題ではない**。
2. ★**時系列交絡の排除（＊偶然だが成立した設計）**：実施順は
   **A(1,2) → B(1..5) → A(3,4,5) → B(6)** ＝**ブロックが時間軸で交互**。
   **A は前半ブロックでも後半ブロックでも 0**／**B は前半でも後半でも成功**
   ⇒ **「時間帯で環境が変わった」では説明できない**（C5 実施47 の時系列交絡の轍を踏まない）。
3. ★**フレッシュ性の強制**：毎セル **(i) scan→`remove`（オブジェクトが在る状態で削除＝`DEL` 確認）**、
   **(ii) DUT 真cold**（by-id 消滅の読み戻し）⇒ **古い bond の再利用ではない**。
4. ★**デバイス側マーカーが «今回発火した» ことを示す**（B run#6・タグで引く）：
   ```
   PAIR 0x60008054 = 0x5dc00000   → タグ 0x5DC0 立=フレッシュ / status=0x00=成功
   ENC  0x60008058 = 0x5de00000   → ENC_CHANGE status=0x00=暗号化成功
   CONN 0x600080C0 = 0x604e0001   → 接続 1 回（毎起動0クリア＝フレッシュ）
   ```
   （`our_sec=0/peer_sec=0` は**正常**＝`PAIRING_COMPLETE` は鍵保存より前に発火する
   ＝C5 evidence-c5-08 §11 の構造。evidence-c3-09 セル1 の成功例 `0x5dc00000` と**同値**。）

**オンチップ flash `[0x10000..0x50000)` を読み戻して md5 照合**（evidence-c3-08 の型）：

## 5. ★真POR は C3 の RTC STORE を消す（センチネルで実証・本ラウンド初）

プロンプトは「**真 POR が C3 の RTC STORE をクリアするかは «未検証»**」と申し送っていた。
**センチネルで確かめた**（＝「クリアされる/されない」を前提にしない）。

**手続き**：アーム B の bond 成功直後（`PAIR=0x5dc00000` / `ENC=0x5de00000` が**実在**）に
**真cold（by-id 消滅の読み戻しで証明）→ ペアリングを一切せずにマーカーを読む**。

| addr | POR 前 | **POR 後（ペアリング無し）** | 読み |
|---|---|---|---|
| `0x54` PAIR | `0x5dc00000` | **`0xa1020805`** | **bt_shim の intr alloc トレース値へ変わった**＝«今回は pairing 未完了» |
| **`0x58` ENC** | **`0x5de00000`** | **`0x00000000`** | ★**消えた** |
| `0x50` SYNC | `0x5ade51c0` | `0x5ade51c0` | 毎ブート app が書く |
| `0x5C` ADV | `0x0ade5000` | `0x0ade5000` | 毎ブート app が書く |
| `0xC4` advRC | `0xad000000` | `0xad000000` | 毎ブート app が書く |

★**over-determination を潰した上で読む**：SYNC/ADV/advRC/PAIR は**毎ブート書き手が居る**ので
「消えた」も「残った」も**判定に使えない**（＝over-determined）。

（`TOPPERS_C3_BOOT_TRACE` 既定 OFF・`0xABF3` WRITE も未発生＝**実測でこのセルでは誰も書いていない**）。
その ENC が **`0x5de00000` → `0x00000000`** になった ⇒ **真POR は RTC STORE を消す**。

★**副産物＝`0x5DC0` 判別子は C3 で «有効»**（C5 の原理的不能と対照的）：
**PAIR は毎ブート `0xA102…`（alloc トレース）で上書きされる**ので
**`0x5DC0` タグが立っていれば «そのブートで pairing_complete が発火した»**＝
**staleness を私が手で消す必要がない**（app の設計どおり＝`ble_host_smoke.c:173-175` のコメントが正しい）。

## 11. ★★本ラウンド最大の成果＝**「C3 BT は hal でなければならない」は誤帰属だった**

**何ラウンドも続いた結論**（`evidence-c3-03` §5「bond 失敗は **供給** に帰属」→
`target.cmake` の既定を hal に固定 → `HANDOFF §4-4`）は、**コンパイラの取り違えによる誤帰属**だった。

**根拠（すべて本ラウンドの実測）**：

1. ★**A/B で違うのはコンパイラだけ**：供給・ソース・**blob（`libbtdm_app.a`＝`859e8c8e`）は同一バイト**
   （§3.2／§4.4 でオンチップ md5 まで同定）。

★**P_DTR＝「C3 でも `dtr=True/rts=True` open が «非リセット» である」＝40%**

- **低めにした理由**：
  1. ★**機構が別**（上表）＝C5/C6 の実測は**移らない**。
  2. ★**MEMORY の既知**＝「**usbjtag コンソールでは真cold を観測できない（ACM open が DUT をリセット→`rst:0x15`）**」
     ＝**usbjtag 経路では «open するとリセットされる» が既に観測されている**。
     ★**Linux の CDC-ACM は open 時に DTR/RTS を «アサート（True）» するのが既定**
     ⇒ **その «既定の open»＝まさに `dtr/rts=True`** であり、**それが従来リセットを起こしてきた**
     ⇒ **「True なら安全」は C3 では «むしろ反証されている» 側**。
  3. ⇒ **C5/C6 と «逆» になる可能性が実在する**（C3 では False の方が非リセットかもしれない）。
- **0% にしない理由**：**USB-Serial-JTAG の ROM も DTR/RTS を解釈する**ので、
  **状態次第でリセットを回避できる余地は原理的にある**（＝**測る価値がある**）。
- ★**外れたら（＝True が C3 でもリセットしない）**：**真cold の console 観測が可能になり
  観測設計が単純化する**。**ただし §16 のマーカー設計は console を必要としない**ので
  **W1 の結論は変わらない**（＝**この検証は W1 の «前提» ではなく «簡単化» の話**）。
- ★**当たったら（＝True もリセットする）**：**C3 では console で真cold を観測できないことが «確定»**
  ⇒ **§16 のマーカー観測系が «唯一の正解» だと裏付けられる**（＝それも収穫）。

### 17.3 ★計器の設計（★`rst:` signature を判定に «使わない»）

★**`rst:` signature は判定に使えない**——**それが証明にならないことは既に実証済み**
（**C5 では `dtr/rts=False` が «電源を切っていないのに» `rst:0x1 (POWERON)` を出した**）。

★**独立の証拠＝`ping_ok` カウンタの «連続性»**（`net/netif_esp32c3.c:203-225`・`COLD_DIAG` 配下）：
- **1Hz で単調増加**し、**アプリが再起動すれば 0 から数え直す**（関数内 `static`＝BSS 初期化）
  ⇒ ★**「open がアプリを再起動させたか」を «console と無関係に» 数で判定できる。**
- ★**over-determination を潰した**：**この build で `0x60008054` を書くのは
  `netif_esp32c3.c` «ただ1つ»**（実測＝コンパイル対象ソース全数を grep）。
- ★**読みが測定を汚さない**：**読むのは run 終了後の1回だけ**。
  `esptool --before usb-reset` は **ROM download mode へ落とすのでアプリは走らない**
  ⇒ **カウンタを進めない**（evidence-c3-02 の「0クリア後5回read→0のまま」と同型）。

★**«open しただけ» を測定対象にする**＝**open 以外の操作（esptool・電源断）を測定窓に混ぜない。**

**アーム（各2回）**：

| 条件 | open 時の DTR/RTS |
|---|---|
| ★**CTRL** | **open しない**（＝**「連続とは何か」を校正する陽性対照**） |
| **T** | `dtr=True` / `rts=True` |
| **F** | `dtr=False` / `rts=False` |

**手順（1 run）**：真cold（by-id 消滅で証明）→ **40s 待つ**（boot+DHCP+ping 蓄積）→
**条件どおり open して 5s 保持 → close** → **30s 待つ** → **`ping_ok` を1回読む**。

**判定**：**`CTRL` の `ping_ok` を «連続» の基準**とし、
**`T`/`F` がそれと同水準なら «非リセット»／大幅に小さければ «リセットされた»**（＝途中で 0 に戻った）。

### 17.4 ★実測結果 ＝ **`dtr/rts=True` は C3 でも «非リセット»**（★予測 P_DTR=40% は **外れ**）

★**判定に使った «決定的で自己検証的な» 計器＝«連続 open»**：
**バッファされた出力は «1回しか» flush できない**⇒ **2回目以降にも «完全な boot ダンプ» が出るなら
それは «その open が新しい boot を起こした» ことの証明**（＝バッファ説では説明できない）。

| 条件 | open#1 | open#2 | open#3 | 判定 |
|---|---|---|---|---|
| **T**（`dtr=True`/`rts=True`）run#1 | 2221 B・**ESP-ROM 無** | **0 B** | **0 B** | ★**非リセット** |
| **T** run#2 | 1987 B・**ESP-ROM 無** | 199 B・ESP-ROM 無 | 196 B・ESP-ROM 無 | ★**非リセット** |
| **F**（`dtr=False`/`rts=False`） | 2841 B・**ESP-ROM 有**・`rst:0x15 (USB_UART_CHIP_RESET)` | **2838 B・ESP-ROM 有**・同 `rst:0x15` | **2376 B・ESP-ROM 有**・同 `rst:0x15` | ★**毎 open リセット** |

⇒ ★**T：6/6 open で «新しい boot が一度も出ない»**／★**F：3/3 open で «毎回フル boot ダンプ»**。
**F の #2/#3 はバッファでは説明不能**（**2.8KB の boot 出力が3回再生成された**）⇒ **F は open ごとにリセットする。**

★**T は «独立の第2計器» とも一致**：`ping_ok`＝**65**（窓 75s）vs **CTRL＝59**（窓 70s）
＝**連続**（＋6 は窓の差 +5s と整合）。**リセットしていれば ok≈24 になるはず**＝**なっていない**。

★★**∴ C3 でも `dtr=True/rts=True` は非リセット＝C5/C6 と «同じ極性»。**

### 17.5 ★予測の的中/外れ

| # | 予測 | 確度 | 実測 | 判定 |
|---|---|---|---|---|
| ★**P_DTR** | C3 でも `dtr/rts=True` が非リセット | **40%** | ★**非リセット（6/6）** | ★**外れ** |

★**なぜ外したか（＝自分の推論の誤りを特定する）**：私は

### 17.6 ★★言えること／言えないこと（★F の «異常» を隠さない）

★**言える（2計器が一致）**：**`dtr=True/rts=True` の open は C3 をリセットしない。**

  ★**T の結論はこの矛盾の影響を受けない**（**T は2計器とも «非リセット» で一致**）。
