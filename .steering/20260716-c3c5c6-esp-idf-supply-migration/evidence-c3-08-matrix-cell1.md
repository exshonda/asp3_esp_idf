# C3 evidence-08 — マトリクス **セル1（hal × iPhone）準備完了**／★「広告していない」の原因を実測で特定

日付: 2026-07-17 ／ branch: `claude/c5-espidf-supply-migration`
DUT: **ESP32-C3 `<MAC-19>`**（hub `1-6` port1）

---

## 1. (a) ★「なぜ広告していなかったか」＝**実測で特定：ボードは壊れていない**

コーディネータ報告＝「`-p 1` で復電させたが広告が確認できない（scan 10秒で未検出）」。

**私の実測**（★広告していないので read は無害＝ユーザーの試行を潰さない）：

| marker | 値 | boot クリア | 読み |
|---|---|---|---|
| SYNC `0x50` | `0x5ade51c0` | されない | sync 到達 |
| PAIR `0x54` | `0xa1020805` | されない | **未ペア**（`0x5DC0` タグ無し） |
| **ENC `0x58`** | **`0x5de0000d`** | されない | **ENC_CHANGE 到達・status=`0x0d`＝`BLE_HS_ETIMEOUT`** |
| ADV `0x5C` | `0x0ade5000` | されない | adv 試行 |
| **DISC `0xB8`** | **`0x00000000`** | **★される** | **切断イベント無し＝«この起動» の話** |
| **CONN `0xC0`** | **`0x604e0001`** | **★される** | **接続が «この起動» で成立した** |
| advRC `0xC4` | `0xad000000` | されない | rc=0 |

★**`CONN`/`DISC` はアプリが毎起動 0 クリアする**（`ble_host_smoke.c:718-719`）
⇒ **`CONN=0x604e0001` かつ `DISC=0` は «現在の起動で接続され、切断されていない»** を意味する。

**⇒ 結論：ボードは «壊れていない»。既知の ETIMEOUT 病態そのもの**
＝**セントラルが接続 → 暗号化 ETIMEOUT → 切断イベント来ない → リンクを握ったまま広告停止**
（evidence-c3-07 §1.2 で整理した病態．**復旧は電源断のみ**）。

### 1.1 ★★運用上の «重大な» 含意（マトリクス測定の前提を壊す）

**コーディネータが `-p 1` で復電させた «後» に接続が起きている**（＝`CONN` は毎起動クリアなので
この起動の話）。**ユーザーの端末が «自動再接続» してボードを再び吊るした**と考えるのが自然。

⇒ **★端末に登録が残っている限り、電源を入れ直すたびに自動再接続で吊られる**
＝**次のセルの測定が «前のセルの端末» に汚染される**。
⇒ **セル切替のたびに «全端末» で forget させる必要がある**（iPhone だけでなく **Android も**）。
**これはユーザー手順に必須の一行**（§3）。

### 1.2 ★これは «hal × Android = ETIMEOUT» の **3度目の再現**でもある

コーディネータの LP_AON 実測 2/2 に加え、**本ラウンドで（意図せず）もう1件**。
⇒ **ユーザーの記憶「hal × Android = OK」とは依然として矛盾**したまま
（＝コーディネータの指摘どおり **C5/C6 の話なら矛盾しない**）。**セル2で決着させる**。

## 3. (c) ★ユーザー向け手順（**セル1＝hal × iPhone**）

### 3.0 ★★最初に必ず（**これを飛ばすとセルが潰れる**）

1. **★Android で `ASP3-C3-BLE` を «登録解除（forget）»**
   — **iPhone のセルなのに Android を forget する理由**：**Android が自動再接続して
   ボードを吊る**（＝本ラウンドで実際に起きた．§1.1）。**吊られると広告が止まり iPhone から
   一切見えなくなる**。
2. **iPhone でも `ASP3-C3-BLE` を «このデバイスの登録を解除»**（GATT キャッシュ対策）
3. **両端末とも Bluetooth を OFF → ON**

★**「接続＋bond 成功」は古い bond の再利用でも起こる**＝**forget 後のフレッシュ試行でのみ測る**。

### 3.1 iPhone での手順（nRF Connect 等の GATT クライアント推奨）

1. スキャンして **`ASP3-C3-BLE`** に接続
2. サービス **`0xABF0`** を開く（**`0xABF1`〜`0xABF4` の4特性**が見えるはず）
3. **`0xABF1` READ** → **`BT4-OK`**（`42 54 34 2d 4f 4b`）
4. **`0xABF2` subscribe** → 32bit LE カウンタが増える
5. **`0xABF3` WRITE** → 任意値（エラーが返らなければ OK）
6. **★`0xABF4` READ** → **未ペアなので «弾かれる» のが正しい**（insufficient authentication）
7. **ペアリング**（NoInputNoOutput 相当＝確認ダイアログのみ・パスキー入力は出ない想定）
8. **bond 後に `0xABF4` を再 READ** → **`BT4-OK` なら D-2c/D-2d 達成**

### 3.2 ★終わったら «触らない»

**試行が終わったらそのまま放置**（コーディネータが LP_AON を読む）。
★**再接続を繰り返すと ENC マーカが上書きされる**（**最後の1件しか残らない**）。

## 4. (d) ★コーディネータが読むべき STORE とタグ

★**タグで引く**（アドレスだけで引くと共用に足を掬われる＝evidence-c3-07 §1.1 の私の誤り）。

| addr | 名前 | boot クリア | タグ／期待値 |
|---|---|---|---|
| `0x60008050` | SYNC | されない | `0x5ADE51C0` |
| `0x60008054` | **PAIR**（★bt_shim intr トレースと**共用**） | されない | **`0x5DC0<status><our><peer>`**＝ペアリング発火。**`0xA102xxxx` のままなら未発火**。**status=`00` で成功** |
| `0x60008058` | **ENC／WRITE**（★**共用．ENC が後勝ち**） | されない | **`0x5DE0<delta秒><status>`**＝ENC_CHANGE 到達。**status=`00`＝成功／`0d`＝`BLE_HS_ETIMEOUT`／`07`＝`BLE_HS_ENOTCONN`**。`0x7717ccff` なら `0xABF3` WRITE の方 |
| `0x6000805C` | ADV | されない | `0x0ADE5000` |
| `0x600080B8` | DISC | **★される** | `0xD15C<reason>…` 形式で切断・**`0` なら «切断イベント来ず»＝吊られている** |
| `0x600080C0` | CONN | **★される** | `0x604E00xx`＝接続成立（xx=回数） |
| `0x600080C4` | advRC | されない | `0xAD000000`（rc=0） |
| `0x600080BC` | — | — | **★ROM が上書き＝使用禁止** |

★**本ラウンドで 7 本すべてを 0 クリアしてから真cold 起動した**
⇒ **非0 の値は «このセッション» のもの**（stale ではない）。
★加えて**マーカは固有マジック**（`0x5DE0`/`0x5DC0`/`0x604E`/`0xD15C`）を持つので
**POR のゴミが偶然作ることはない**⇒**タグが立っていれば «発火した» と読んでよい**。

### 4.1 成功時（セル1 が «OK» なら）

```
SYNC = 0x5ADE51C0 / ADV = 0x0ADE5000 / advRC = 0xAD000000
CONN = 0x604E00xx           ← 接続
ENC  = 0x5DE0xx00           ← ★status=00 = 暗号成功
PAIR = 0x5DC000xy           ← ★status=00 = bond 成立
```

### 4.2 ユーザー仮説どおり «NG» なら（予想される形）

```
CONN = 0x604E00xx / ENC = 0x5DE0xx<非0>  ← status が 0 以外（0d/07 等）
PAIR = 0xA102xxxx            ← 未発火
DISC = 0 なら «吊られている»（→ 次セルの前に必ず -p 1 で電源断）
```

★**読むと DUT が download mode に落ちて広告が止まる**（`--after no-reset`）
⇒ **読了後に次セルへ行くなら `-p 1 -a off/on` で真cold から**。

## 5. (e) 残ブロッカー

1. **★セル2 以降は «焼き直し» が要る**（セル3/4＝esp-idf 供給）。**コーディネータの合図待ち**
   （＝ユーザー試行 → 読み → 電源断 → 次、の順）。**先走って焼かない。**
2. **★`ENC` マーカは «最後の1件» しか残らない**⇒**1ビルドにつき1端末**。
   **他 central を挟ませない**（コーディネータの制約 #1）。
3. **★自動再接続の罠**（§1.1）＝**全端末で forget させないと、電源を入れ直すたびに吊られる**。
   **これはマトリクス全セルに効く前提条件。**
4. **セル2（hal × Android）は «再現の取り直し»**＝**私とコーディネータの実測（ETIMEOUT）と
   ユーザーの記憶（OK）が矛盾**している。**本ラウンドで（意図せず）3度目の ETIMEOUT を観測**した
   ので、**矛盾は «ユーザーの記憶が C5/C6 の話» で説明できる可能性が高い**——**が、断定しない**。
