## 2. 計装（既存資産の再利用＋最小追加．**passive**）

`bt/evt_trace.c`（`ESP32C3_BT_EVT_TRACE=ON`．既定OFF）は既に
**NimBLE ホストが処理する «全» HCI イベントの чоkepoint**
`ble_hs_hci_evt_process()` を `--wrap` している（`ble_hs.c` からの**クロスTU**呼出し
＝`--wrap` が確実に効く）。ACL 側も `ble_mqueue_put`／`ble_mqueue_get`／`ble_l2cap_rx` を wrap 済み。

**追加したのは «HCI ステージ地図» の出力先だけ**（既存の RTC STORE0 パッキングは無改変）：

- **出力先＝RTC FAST RAM `0x50000100`〜**（20 語）。
  ★**実測**：`0xCAFEBABE` を書いて `--before usb-reset` を伴う read-back で
  **usb-reset を跨いで保持されることを確認**（RTC STORE は 7 語しかなく
  app の SYNC/ADV/CONN/DISC/PAIR と食い合うため、語数の要る地図は FAST 側へ）。
  **POR では消える**ので cleared-boot-read がそのまま効く。
  （app は既に `0x50000080` を DHCPDIAG に使用＝スクラッチとしての前例あり。）

| idx | addr | 内容 |
|---|---|---|
| 0 | `0x50000100` | magic `0x5C3E0001`（地図が書かれた証拠） |
| 1 | `0x50000104` | HCI EVT 総数（host 処理） |
| 2 | `0x50000108` | **LE Connection Complete**（0x3E/0x01）数 |
| 3 | `0x5000010C` | **Disconnection Complete**（0x05）数 |
| 4 | `0x50000110` | **直近 Disconnection の reason** |
| 5 | `0x50000114` | LE LTK Request（0x3E/0x05）数 |
| 6 | `0x50000118` | Encryption Change（0x08）数 |
| 7 | `0x5000011C` | 同 status |
| 8/9/10 | `0x50000120`/`24`/`28` | **ACL: put / get / l2cap**（host rx_q 到達／drain／L2CAP 到達） |
| 11/12/13 | `0x5000012C`/`30`/`34` | Command Complete / Command Status / その他 LE Meta |
| 14-19 | `0x50000138`〜 | **到着順の EVT code 24個**（4個/語．0x3E は `0xE|subevent` へ畳む） |

★★**「blob だけ差し替える第3アーム」はやらない**——**それ自体が «混成»** であり
ABI 不整合と機能差を分離できない（私自身の警告．C6 evidence-c6-05 と同型）。
**本計装は A/B の «同一コード» に同じ窓を開けるだけ＝混成を作らない。**

## 3. ★★事前登録予測（実機前）

### 3.0 ★まず «計装が測定を壊していないか» の対照（これが先）

**`--wrap` は 4 シンボルを差し替える＝ビルドが変わる。**
∴ **B（hal）＋トレース ON で «bond が依然成功する» ことを先に確認する。**

| # | 予測 | 確度 |
|---|---|---|
| **C0** | **B＋トレースON で bond 成功**（＝計装は非破壊） | **85%** |

★**C0 が外れたら（B がトレースONで落ちる）、A の地図は «計装の副作用» と区別できない**
⇒ **その場合は機構を主張せず「計装が侵襲的だった」と報告して終える**（無理に成功を作らない）。

### 3.1 機構の予測

| # | 予測 | 確度 | 根拠 |
|---|---|---|---|
| **M1** | **A も `conn_cmpl ≥ 1`**（host は接続を見ている） | **75%** | A run#2 で app の CONN コールバックが発火（`0x604e0001`）＝LE Conn Complete は少なくとも時々届く。∴ «接続前» で死ぬ説は劣勢 |
| **M2** | **A は `ltk_req = 0` かつ `enc_chg = 0`** | **85%** | bond は暗号開始まで到達していない（BlueZ 側が pairing を «キャンセル» している＝30s タイムアウトではなく早期終了） |
| **M3** | **A は `acl_l2cap = 0`、B は `acl_l2cap > 0`**（＝SMP PDU が L2CAP まで来ない） | **45%** | blob 差が ACL RX 配送に出る説。★**45%＝«分からない» に近い**。対抗仮説：L2CAP までは来るが SMP 応答が不正 |
| **M4** | **A は `disc ≥ 1` で reason ≠ 0** | **70%** | central が pairing を諦めた後に切断するはず。**reason の値が機構の手がかり**になる |

### 3.2 ★判別表（**A と B の «差分» でのみ読む**．A の絶対値では読まない）

| A の地図 | B の地図 | 読み |
|---|---|---|
| `conn_cmpl=0` | `conn_cmpl>0` | **LE Conn Complete が host へ届かない**＝controller→host の EVT 経路 |
| `conn_cmpl>0, acl_put=0` | `acl_put>0` | **接続後の ACL RX を controller が配送しない**＝blob の ACL 経路 |
| `acl_put>0, l2cap=0` | `l2cap>0` | **host 側が L2CAP 前に落とす**（conn_find NULL 等） |
| `l2cap>0, ltk_req=0` | `ltk_req>0` | **SMP は流れるが暗号開始に至らない**＝SMP 応答内容の問題 |
| **両者ほぼ同一** | — | **★HCI 層では差が出ない**＝差は «HCI より下»（LL/RF）か «より上»（SMP 内部状態）＝**本計装では決まらない**と報告する |

### 3.3 ★含意の自問（「反証条件も仮説である」）

| | 主張 | 健全か |
|---|---|---|
| (a) | 「A で X=0 ⇒ バグは X の地点」 | **★健全でない**＝**B が同じ地点で X>0 を示して初めて «差» になる**。∴**必ず A/B 両方を測る**（A 単独の地図は解釈不能） |
| (b) | 「地図に差が出ない ⇒ 供給は無関係」 | **★健全でない**＝evidence-c3-03 の 2/2 vs 2/2 は**既に帰属を確定させている**。差が出ないなら「**HCI 層では見えない**」だけ＝**帰属は生きたまま機構が未特定**と書く |
| (c) | 「`disc reason` が X ⇒ 原因は X」 | **★健全でない**＝reason は **central が諦めた «結果»** かもしれない（原因ではなく症状）。**順序（seq）と併せてしか読めない** |
| (d) | 「C0（B＋trace で bond 成功）⇒ trace は完全に無害」 | **健全でない（が実用上十分）**＝B で無害でも A で侵襲的な可能性は残る。**ただし A/B 同一ビルド条件なので «差» の解釈には耐える** |

### 3.4 測定手順（固定）

各アームで：**`bluetoothctl remove`（フレッシュ化）→ RTC FAST を 0 クリア → 0 を検証 →
真cold（`-a off` ＋ by-id 読み戻し 0）→ 起動 →★**スキャン/pair を先に**（`read-mem` は
DUT を download mode へ落とすので**最後**）→ 地図を読む**。
★evidence-c3-03 §5.3-1 で **「採取が DUT を止める」罠を実際に踏んだ**ので順序を固定する。
