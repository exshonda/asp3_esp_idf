## 2. 計装（既存資産の再利用＋最小追加．**passive**）

`bt/evt_trace.c`（`ESP32C3_BT_EVT_TRACE=ON`．既定OFF）は既に
**NimBLE ホストが処理する «全» HCI イベントの чоkepoint**
`ble_hs_hci_evt_process()` を `--wrap` している（`ble_hs.c` からの**クロスTU**呼出し
＝`--wrap` が確実に効く）。ACL 側も `ble_mqueue_put`／`ble_mqueue_get`／`ble_l2cap_rx` を wrap 済み。

**追加したのは «HCI ステージ地図» の出力先だけ**（既存の RTC STORE0 パッキングは無改変）：

- **出力先＝RTC FAST RAM `0x50000100`〜**（20 語）。
  ★**実測**：`0xCAFEBABE` を書いて `--before usb-reset` を伴う read-back で
  **usb-reset を跨いで保持されることを確認**（RTC STORE は 7 語しかなく
  app の SYNC/ADV/CONN/DISC/PAIR と食い合うため、語数の要る地図は FAST 側へ）。
  **POR では消える**ので cleared-boot-read がそのまま効く。
  （app は既に `0x50000080` を DHCPDIAG に使用＝スクラッチとしての前例あり。）

| idx | addr | 内容 |
|---|---|---|
| 0 | `0x50000100` | magic `0x5C3E0001`（地図が書かれた証拠） |
| 1 | `0x50000104` | HCI EVT 総数（host 処理） |
| 2 | `0x50000108` | **LE Connection Complete**（0x3E/0x01）数 |
| 3 | `0x5000010C` | **Disconnection Complete**（0x05）数 |
| 4 | `0x50000110` | **直近 Disconnection の reason** |
| 5 | `0x50000114` | LE LTK Request（0x3E/0x05）数 |
| 6 | `0x50000118` | Encryption Change（0x08）数 |
| 7 | `0x5000011C` | 同 status |
| 8/9/10 | `0x50000120`/`24`/`28` | **ACL: put / get / l2cap**（host rx_q 到達／drain／L2CAP 到達） |
| 11/12/13 | `0x5000012C`/`30`/`34` | Command Complete / Command Status / その他 LE Meta |
| 14-19 | `0x50000138`〜 | **到着順の EVT code 24個**（4個/語．0x3E は `0xE|subevent` へ畳む） |

★★**「blob だけ差し替える第3アーム」はやらない**——**それ自体が «混成»** であり
ABI 不整合と機能差を分離できない（私自身の警告．C6 evidence-c6-05 と同型）。
**本計装は A/B の «同一コード» に同じ窓を開けるだけ＝混成を作らない。**

## 5. ★★実機結果 — **C0 が «外れて» 私の計装が侵襲的だと露呈した**（そして直した）

### 5.1 ★C0（計装の非破壊性）— **最初は外れた**

| # | 構成 | bond |
|---|---|---|
| **C0a** | hal ＋ トレース ＋ **私の 20 語 map** | **失敗**（`AuthenticationCanceled`）＝**A と同じ症状** |
| **C0b** | hal ＋ **既存の evt_trace のみ**（map 無し） | **成功**（`Pairing successful`） |
| **C0（再）** | hal ＋ **軽量化した map** | **成功**（`Pairing successful`） |

★**C0a が «A と同じ失敗» を出した**＝**もし C0 を測らずに A/B だけ比べていたら，
«計装が作った失敗» を «供給の失敗» と読んでいた**（＝偽の機構物語を恒久記録に焼き付けた）。
**§3.0 で «C0 が外れたら機構を主張せず終える» と先に宣言していたので踏み止まれた。**

**原因（実測で分離）**：**私の map が hot path で 20 語 dump していた**。
`evt_trace_pack()` は `ble_mqueue_put`／`ble_mqueue_get`／`ble_l2cap_rx`＝**ACL 毎パケット**から
呼ばれる。既存実装はそこで **1 語**しか書かない（だから無害だった）。**私はそこに 20 語を足した**
⇒ SMP のタイミングが壊れた。**C0b が «既存facility は無罪» を示す**＝**私の追加が犯人**と確定。

**修正**：**dump を HCI EVT wrap からのみ呼ぶ**（pairing 中せいぜい十数回）。
**ACL hot path からは呼ばない**。⇒ **C0 再測で成功**＝計装の非破壊性を実証してから A/B を測った。

> ★**副次的な «道具» の教訓**：**既存の evt_trace は «動く構成» に対して一度も検証されていなかった**
> （作られた当時は bond が失敗していたので，侵襲的でも気づけない）。**C0 のような
> «健全な対照に計装を当てる» 手順が無いと，計装の侵襲性は永遠に発見されない。**

### 5.2 ★★A/B の HCI ステージ地図（**C0 通過後・全て真cold・同一手順**）

| 項目 | **A**（esp-idf／真の v5.5.4） | **B**（hal） |
|---|---|---|
| bond | **失敗**（`AuthenticationCanceled`） | **成功**（`Pairing successful`） |
| HCI EVT 総数 | **13** | **28** |
| **LE LTK Request**（0x3E/0x05） | **0** | **1** |
| **Encryption Change**（0x08） | **0** | **1**（status **0**＝成功） |
| ACL **put / get / l2cap** | **6 / 6 / 6** | **17 / 17 / 17** |
| Disconnection Complete | **0** | **0** |
| 到着順 seq0/1/2 | `0ce7e4ea` `1313e7e4` `13131313` | **完全に同一** |

**A の全13イベント（デコード）**：

```
[0] 0xEA LE Enhanced Connection Complete   [4] 0xE4 LE Read Remote Features
[1] 0xE4 LE Read Remote Features            [5] 0xE7 LE Data Length Change
[2] 0xE7 LE Data Length Change              [6-12] 0x13 Number Of Completed Packets ×7
[3] 0x0C Read Remote Version Complete
```

### 5.3 ★機構の局在（依頼 (a)(b)）— **ここまでは決まった／ここから先は決まっていない**

**決まったこと（A/B の «差» として．PROVEN）**：

1. **接続確立は A と B で «完全に同一»**（seq が**バイト一致**＝Enhanced Conn Complete →
   Read Remote Features → Data Length Change → Read Remote Version …）
   ⇒ **接続前／接続確立で死ぬ説は «反証»**。
2. **host 側 RX パイプラインは A でも健全**：**`put == get == l2cap`（6=6=6）**
   ⇒ **ACL は host まで届き，drain され，L2CAP まで回っている＝host は1個も落としていない**
   ⇒ **「host が落としている」説も «反証»**（＝旧 evt_trace が想定した put>get／get>l2cap の型ではない）。
3. **A は L2CAP を 6 パケット処理した «後» に停止し，LTK Request（暗号開始）に一度も到達しない**
   （B は 17 パケット処理して LTK Request → Encryption Change status=0 まで到達）。
4. **A の側から見て切断イベントは発生していない**（disc=0）＝**central が諦めるまでリンクは生きている**。

⇒ **§3.2 の事前登録判別表の «`l2cap>0, ltk_req=0` vs `ltk_req>0`» の行に該当**：
   **「SMP は流れるが暗号開始に至らない＝SMP 交換の «中身» の問題」**。

**∴ 機構は «SMP 交換の内部»（接続確立の後・暗号開始の前）に局在した。**
**HCI トランスポート／ACL 配送／host の L2CAP ルーティング／接続確立は «すべて無罪»。**

**決まっていないこと（正直に）**：

- **本計装は RX しか数えていない**（`ble_l2cap_rx`＝受信側）。
  ⇒ **ペリフェラルが «応答を返していないのか» «誤った応答を返しているのか» は区別できない**。
- **6 パケット目が «どの SMP PDU» かは分からない**（CID/opcode を見ていない）。
- ∴ **「SMP のどの手順で」までは絞れていない**。**そこは次ラウンドの仕事**。

- (a)「A で X=0 ⇒ バグは X」＝**不健全と宣言済**。実際 **`conn_cmpl=0` は B でも 0**（計器バグ）

  ⇒ 宣言が**そのまま役に立った**。
