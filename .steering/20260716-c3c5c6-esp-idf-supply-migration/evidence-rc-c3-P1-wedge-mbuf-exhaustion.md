計器＝`p1-tools/c3_postmortem.sh`＋`c3_postmortem_decode.py`（**出荷ビルドに計装ゼロ・JTAG 読み出しのみ**）。
生ダンプ＝`p1-tools/dump_DEAD_wedge.txt`・`dump_DEAD_leaked_mbufs.txt`、対照＝`sample_healthy_dump.txt`。

## 0. 経過（実機・Android・同日）

同一 build（本セッションの修正入り＝`t2c5_c3_ble`／esp-14.2.0_20260121／SM=ON）で：
**ペアリング成功 → 再接続も成功 → 切断/再接続を «何回か» 繰り返した後、connect できなくなった

### 1-4 ★★★ 枯渇した12ブロックの正体＝**未解放の notify 送信キュー**
12ブロック全部が**完全に規則的な対**（`dump_DEAD_leaked_mbufs.txt`）：
- 偶数ブロック：`om_len=4`・`om_next=0`・pool=msys_1 ＝ **notify の4バイト・ペイロード**
- 奇数ブロック：`om_pkthdr_len=8`・`om_len=7`・`om_next`→直前の偶数ブロック・pkthdr 総長 `0x0b=11`
  ＝ **ATT Handle-Value-Notification パケット**（L2CAP 4 + ATT 7 = 11B）

⇒ **msys_1 の全12ブロック＝ちょうど «notify 6パケット»（ヘッダ mbuf＋ペイロード mbuf の対）が丸ごと残留。
リークは 100% «送信キューに積まれた notify» に帰属する。**

`notify_tick` は **1秒周期**（`ble_host_smoke.c:248`）＝**リンク死から約6秒で12ブロックを使い切る**。

## 2. 解釈（★事実と推測を分ける）

### 2-1 **wedge の機構は特定できた**（確信度：高い＝上記実測の直結）
1. リンクが死ぬ（supervision timeout）
2. **切断イベントが host に配送されない（DISC=0）**——今回 5接続/4切断で**最後の1回だけ**欠落
3. host は接続の後片付けをしない ⇒ **送信キューの mbuf が解放されない**
4. `notify_tick` が1秒毎に積み増し、**約6秒で msys_1(12ブロック) を使い切る**
5. 以後 host は**何も確保できない**（notify 117連続失敗）⇒ **広告も再開できない**
   （`g_adv_active=1` なのに電波が出ていない）⇒ **スマホから見えない＝「connect できない・エラーも出ない」**

★**「エラーが出ない」というユーザー観察は、この機構の予測どおり**（デバイスが広告しないので、
スマホは «見つからない» だけでエラー事象が発生しない）。

### 2-2 **まだ分かっていないこと**（正直に）
- ★**最初にリンクが死んだ理由は未解明**。今回の枯渇は «リンク死の後» に notify が積んだ結果であり、
  **リンク死の原因ではない**（12ブロック＝6秒分は死後に消費されている）。
- ただし**「前サイクルの残留で入口から枯渇していた」可能性は排除できない**
  （今回は最後の1接続だけで枯渇を説明できるので、**cross-cycle 累積は不要**だが、否定もしていない）。
- `shim_que_pend_used=71` の内訳（どの DTQ か）は未取得。

## 3. ★出荷可能な成果（計画 §8 のオフランプが現実になった）

**wedge の «reset 不要化» が設計可能**：host が «接続していると信じているのに送れない» 状態を
アプリ側で検出し（例：notify 連続失敗 N 回 or 一定時間 notify 成功なし）、`ble_gap_terminate` を
撃って接続を畳めば、**NimBLE が接続の後片付けで送信キューの mbuf を解放する**はず。
⇒ 広告が再開し、スマホから再び見えるようになる（**reset 不要**）。
※「はず」＝**未検証**。次段で実測する（P1-3b）。

## 4. 次の一手

1. **リーク/枯渇の因果を分離**：リセット後、接続/切断サイクルごとに `mp_num_free` を JTAG で読む。
   - 単調減少して戻らない ⇒ サイクル毎リーク（＝入口枯渇もあり得る）
   - 各サイクルで戻る ⇒ 今回のように «死んだ接続» でのみ滞留
2. **P1-3b（wedge 解消の実証）**：watchdog で `ble_gap_terminate` → `num_free` が回復し広告が戻るか。
   戻れば**出荷可能な回避策**が確定（根治前でも UX が直る）。
3. **リンク死の原因**（残る本丸）は P2（stock 成功 vs 我々 失敗の同一スマホ btsnoop 差分）へ。

## 5. 計画への反映
- **H5（RTOS/shim デッドロック）は反証** ⇒ 計画 §3 を更新すること。
- **DISC=0 は «独立軸» ではなく wedge の «起点»** と確定（fable 指摘6の読みが実測で支持された）。
- 計器（`p1-tools/`）は**計装ゼロで再利用可能**＝以後の全ラウンドで同じ手が使える。

## 6. ★事前登録（P1-A・測定 «前» に記録）

### 6-1 これまでの実測（サイクル1＝陰性対照）
- T0(fresh boot)：`msys_1 free=12/12 min=12`・`conn=0xffff`・notify 0/0・gap 0/0
- C1（**notify 無し**・クリーン切断）：`free=12/12 min=10`・**gap_conn=1 / gap_disc=1**（切断は届いた）
  ⇒ **クリーンな切断ではリークしない**（一時的に2個使い、正しく解放）。**陰性対照として成立**。
- ユーザー証言：死亡時の接続は **nRF Connect で Notify を有効化**していた
  （＝`g_notify_enabled=1`・`notify_sent=24` と整合）。

### 6-2 ★予測（この通りに動けば機構は確定、外れれば仮説を捨てる）
**H6：リークは «notify が流れている» ＋ «切断イベントが失われた» の両方が揃った時だけ起きる。**

| 条件 | 予測される観測 |
|---|---|
| **P1**：notify ON ＋ **クリーン切断**（gap_conn == gap_disc） | `free` は **12/12 に復帰**。`min_free` は notify 分だけ下がるが戻る。**リークしない** |
| **P2**：notify ON ＋ **切断イベント欠落**（gap_conn > gap_disc） | `free` が **0/12** まで落ちる（notify 1秒周期 × 6パケット＝**約6秒で枯渇**）。`conn_handle != 0xffff` のまま。`notify_fail` が増え続ける。**広告停止＝「connect できない・エラー無し」** |
| **P3**：notify **OFF** で切断イベント欠落 | `conn_handle` は残る（wedge）が **`free` は 12/12 のまま**（リーク経路が動かない）＝ **wedge と枯渇は別事象**であることの分離 |

**反証条件**：notify ON のクリーン切断で `free` が戻らない（＝P1 が外れる）なら、
「クリーン切断でも解放されない」ことになり H6 は誤り。
また gap_conn > gap_disc なのに `free=12/12` のままなら（P2 が外れる）、
枯渇は切断欠落と無関係となり H6 は誤り。

### 6-3 測定手順
nRF Connect で接続 → **Notify を ON** → 20秒保持 → 切断 →（切断中に）`c3_pool_watch.sh` で読む。
これを繰り返し、`gap_conn`/`gap_disc` の一致・不一致で P1/P2 のどちらのセルかを判定する。

## 7. P1-A 結果 — ★**事前登録した予測 P2 は反証された**（＝機構がむしろ明確になった）

### 7-1 実測（`p1-tools/c3_pool_watch.sh`・全て切断中に測定）
| 時点 | msys_1 free(min) | conn_handle | notify sent/fail | gap conn/disc |
|---|---|---|---|---|
| T0 fresh boot | 12/12 (12) | 0xffff | 0/0 | 0/0 |
| **C1**（notify 無し・クリーン切断） | **12/12 (10)** | 0xffff | 0/0 | **1/1 ＝届いた** |
| **C2**（nRF Connect・**notify ON**） | **12/12 (10)** | **0x0001** | 52/0 | ★**2/1 ＝欠落** |
| C2 +4〜+24s（連続観測） | **12/12 (10) 不変** | 0x0001 | **81→97 / 0** | 2/1 |
| リセット後 | 12/12 (12) | 0xffff | 0/0 | 0/0 |

### 7-2 ★予測 P2 の反証（事実）
事前登録（§6-2）では「**notify ON ＋ 切断イベント欠落 → 約6秒で `free`=0 まで枯渇**」と予測した。
実測は **24秒間・notify を 1秒毎に送り続けても（sent 52→97）`free`=12/12・`fail`=0＝リークゼロ**。
⇒ **H6（「notify 流通＋切断欠落」で枯渇する）は誤り。** 相手が居なくても mbuf は正しく解放される。

**帰結**：初回死亡時の枯渇（sent=24 で `free`=0・fail=117）には**別の条件**が必要。
**推測（未検証と明示）**：死亡時は**コントローラが送信完了を返さなくなり** host の送信キューが滞留した。
⇒ **枯渇は «コントローラ停止» の下流症状**であって、DISC=0 の直接の帰結ではない。

### 7-3 ★障害レベルの分離（これが本節最大の成果）
| レベル | 条件 | 症状 | プール |
|---|---|---|---|
| **L1** | 切断イベント欠落 → host が接続を保持 → **広告を出さない** | ★**スマホから見えない＝「connect できない・エラー無し」** | **健全** |
| **L2** | L1 ＋ コントローラが送信完了を返さない | 上記＋host が何も確保できない（完全 wedge） | **枯渇** |

★**ユーザー可視の主症状は L1 だけで完全に再現する**（実測：C2 の状態でユーザーのスマホから
`ASP3-C3-BLE` が「見えない」ことを確認）。**mbuf 枯渇は主症状の必要条件ではない。**
⇒ **修正の狙いが L1（stale 接続の検出と解消）に絞れた**。枯渇対策は L2 の緩和策。

※PC 側 BlueZ スキャナはこの時点で全デバイス検出 0 になり**陽性対照が取れなくなった**ため、
広告有無の判定は**ユーザーのスマホ（nRF Connect＝実績のある計器）**で行った。
「スキャナが死んでいる状態での『見えない』は証拠にならない」を適用した結果である。

### 7-4 新しい手がかり H7（★交絡あり・次で切り分ける）
| サイクル | クライアント | notify | 切断イベント |
|---|---|---|---|
| C1 | （設定画面と推定） | OFF | **届いた** |
| C2 | nRF Connect | **ON** | ★**欠落** |

**H7：notify のトラフィックが切断イベントの配送を失わせている**（送信経路とイベント経路の競合）。
⚠️**C1/C2 はクライアントも違う＝交絡**。次サイクルは **nRF Connect のまま notify を OFF** にして
同一クライアントで notify だけを変える（交絡排除）。
- 切断が届く ⇒ **notify が犯人**
- 切断が失われる ⇒ notify は無関係（クライアント/接続の仕方など別要因）

## 8. ★★★ P1-3b — **DISC=0 の «層» が確定：コントローラ自身がリンク喪失を検出していない**

### 8-1 実験
アプリに接続 stale watchdog（`ESP32C3_BT_CONN_WD`・既定 OFF＝出荷ビルド不変）を追加。
接続保持中は毎秒 **HCI Read RSSI をコントローラへ**発行し、rc と RSSI を記録する
（当初設計：rc≠0＝コントローラが当該 handle を知らない＝host だけ stale と判定して terminate）。
この probe は **修正であると同時に «DISC=0 の層» を決める計測**として事前に位置づけた
（rc≠0 ⇒ イベント配送だけが落ちている／rc==0 ⇒ コントローラも死に気づいていない）。

### 8-2 実測（ペアリング失敗・再接続不可・スマホから不可視の状態で）
| 値 | 観測 |
|---|---|
| `g_wd_probe_count` | 57 → 61 → 66（**1秒毎に稼働**） |
| **`g_wd_probe_last_rc`** | ★**0（成功）** |
| **`g_wd_last_rssi`** | ★**-50 dBm**（もっともらしい実値） |
| `g_wd_term_count` | 0（＝設計通り「生きている」と判断し畳まなかった） |
| `g_conn_handle` / `gap_conn` / `gap_disc` | 0x0001 / 1 / **0** |
| msys_1 | 12/12（min=11）＝**枯渇なし**（L1 のみ） |

**ユーザー証言（同時刻）**：**「一回も接続が成功していないので切断状態」**＝**スマホ側にリンクは無い**。

### 8-3 ★結論（確信度：高い＝両端点の突合）
> **コントローラは、既に存在しないリンクに対して `HCI Read RSSI` に rc=0 と RSSI=-50 を返す。**
> ⇒ **コントローラ自身がリンク喪失（supervision timeout）を検出していない。**
> ⇒ 切断イベントは **«host が取りこぼしている» のではなく «そもそも生成されていない»**。

**これが長年の不変量 DISC=0 の正体であり、層は «コントローラ／LL の supervision 検出» と確定する。**
「host／shim のイベント配送欠落」説はこれで**否定**された（§1 の RTOS/shim 健全と併せて二重に）。

### 8-4 副産物：**RSSI は stale 検出器として使えない**（設計の訂正）
コントローラは死んだリンクに対して**嘘の成功**を返すため、当初の watchdog トリガは無効。
⇒ トリガを **「接続保持中に GAP 事象が一定時間まったく動かない」**（`WD_IDLE_LIMIT=45s`）へ変更。
RSSI は **«コントローラが嘘をつく» ことの証拠**として診断値のまま記録し続ける。

### 8-5 計画への反映
- 計画 rev2 の «未解明の核心 A（なぜ OTA リンクが死ぬか）» は、より正確には
  **«なぜコントローラが supervision timeout を検出しないか»** に書き換わる。
- 対立仮説 H2（アンカードリフト）／H3（PHY 劣化）は、**「リンクが実際に切れているのに
  コントローラが気づかない」**という観測と整合的に再評価する必要がある
  （＝コントローラの時間基準／接続イベント計数が進んでいない可能性）。
- ★次の最有力の問い：**コントローラの LL 時間基準（接続イベントの計数／supervision カウンタ）が
  我々の統合下で進んでいないのではないか**。C5/C6 が同型の shim で成功していることとの差分が鍵。

## 9. ★P1-3b 実証 — **自動復帰（reset 不要化）が成立**（2サイクル連続）

### 9-1 実装（`ESP32C3_BT_CONN_WD`・既定 OFF＝出荷ビルド不変）
2段構え。トリガは実測に基づいて2回訂正した：
1. **1段目**：接続保持中に **GAP 事象が 45秒途絶** → `ble_gap_terminate()`
   （※当初の「RSSI 失敗で検出」は**コントローラが嘘の成功を返すため無効**と実測 → 変更）
2. **2段目**：terminate 後 **10秒**たっても接続が残る → ★**`ble_hs_sched_reset()`**
   （※`ble_gap_terminate()` は **rc=0 を返すのに完了しない**（Disconnection Complete が
   永久に来ない）と実測 → エスカレーションを追加）
   → ホスト再構築 → `on_sync()` が `start_advertising()` を呼ぶ → **広告復帰**

### 9-2 実測（Android・nRF Connect）
| サイクル | 経過 | 実測 |
|---|---|---|
| 1 | 接続→**bond 成功**→勝手に切断→放置 | `term=1` → **`reset=1`** → `conn_handle=0xffff`・`gap 1/1`・msys_1 12/12 |
| — | ユーザー確認 | ★**「見えた，接続したら connected/bond になった」＝自動復帰を実機で確認** |
| 2 | 再接続→**bond 成功**→即切断→放置 | `term=1` → **`reset=2`** → `conn_handle=0xffff`・**`gap 2/2`（欠落ゼロ）**・msys_1 12/12 |

★**2サイクル連続で、ユーザー操作なしに約55秒で復帰**（45s + 10s）。
復帰後は**再接続も bond も成功**する。帳尻（`gap_conn == gap_disc`）も合う。

### 9-3 到達点と限界（★正直に）
- ★**達成**：「一度失敗すると reset するまで二度と繋がらない」という**最悪 UX を解消**。
  出荷可能な緩和策として成立（既定 OFF なので有効化は選択制）。
- ★**未達（本丸）**：**リンクが bond 直後に死ぬ根本原因は未解明のまま**。
  今回の実測でも「接続→bond 成功→直後に死ぬ」が 2/2 で再発しており、
