## 3. ★予測（**実機を触る前に固定。測定後に改竄しない**）

### 3.1 前ラウンドの予測を**引き継がない＝反転させる**

evidence-c6-01 §7.1 の予測は「**ハングする**（blob 起因）」。
**私はこれを引き継がず、「動作する（＝グルー起因／blob は無罪）」側に賭ける。確信度 ≈ 65%。**

★これは前ラウンドの記録の**改竄ではない**。前ラウンドの予測は `22ea5a2` に確定済みで、
本節は**それとは別に、本ラウンドの実施者が測定前に登録する予測**である。両方を答合せする。

### 3.2 反転の根拠（重み順）

1. **★「blob 起因」を支える最大の証拠（§13 の v6.1 swap で直った）は、まさに今回解こうとしている
   交絡そのもの**。glue も同時に替わっているので、**§13 は blob 説に独立な重みを与えない（≈0）**。
   ここを差し引くと、blob 説の証拠基盤は §12 だけになる。
2. **§12 の「発散点は `register_chipv7_phy` 内部」は under-determined**。§12 が実測したのは
   **hal グルーの中での BT vs WiFi 比較**であり、入力（a0-a3／`init_data` 128B／`cal_data`）3/3 一致・
   **エントリ MMIO «11本» 一致**。しかし modem/PHY のレジスタ空間は 11本では尽きない。
   ∴「入口の状態は同一」は**11本の範囲でしか成立していない**。**そして §12 は「C3型グルーなら
   どうなるか」を一切測っていない**（hal グルーしか存在しなかったため）。
3. **★hal グルーは «まさにこのクロック設定の領域» で実際に2件のバグが実証済み**（§11）：
   (i) regi2c マスタのクロック源未選択（`sel_160m=0`／WiFi は両方呼ぶが BT の
   `esp_shim_bt_clock_init()` は ICG だけ）、(ii) `WIFIPWR` ビット欠落。
   ＝**「hal の BT グルーは WiFi がやっている PHY 前段セットアップを取りこぼす」という前例が
   既に2件ある**。3件目（synth に効くもの）が在る事前確率は低くない。**C3型 `bt.c` は上流が
   サポートする正規のセットアップ経路**であり、取りこぼしを埋める側。
4. **★C5 の構造的に同型な前例**（本日・別エージェント）：記録「実施09＝hal(v8) libphy は BT 経路で
   `register_chipv7_phy` が収束しない」に対し、**hal とバイト同一の blob（v5.5.4タグ）× C3型グルー**で
   **真cold D-1 2/2＋W2（OTA GATT）達成**＝**「hal 相当 blob だから収束しない」は C5 では成立しなかった**。
   C6 も同じ形（記録＝blob 帰属／未実施＝4アーム目）。
   ※**射程限定**：C5 の結果は C6 のシリコン/blob には移送できない（チップが違う）。
   これは「blob 説が唯一の説明ではない」という**存在証明**としてのみ使う。

### 3.3 「ハング」側に残る根拠（＝私が負ける筋。正直に書く）

- §12 の入力3/3一致は事実であり、**もし 11本の MMIO が実質的に十分だったなら**、内部分岐説が正しい。
- `libphy.a` は v5.5.4(=hal) と v6.1 で**実際にバイトが違う**＝版差は実在する。
  「BT サブパスだけが v6.1 で直った」は物理的にあり得る話であり、**C5 の実施09 も
  «libphy 版数不適合» が真因だったと記録されている**（ただし C5 は WiFi も非収束＝C6 と型が違う）。

### 3.4 ★反証条件（**含意が本当に成立するかを自問してから書く**）

親プロンプトの表は「動作→グルー起因／ハング→blob 起因」だが、**この含意をそのまま機械適用しない**
（C5 §10.2 の失敗＝事前登録した反証ルール自体が invalid だった、の再発を避ける）。
**含意を «直接の観測量» に限定して書く**：

| 実機結果 | **一意に言えること（強い）** | **言えないこと（＝含意を連鎖させない）** |
|---|---|---|
| **★動作（D-1 到達）** | **hal と同一バイトの blob が C6 実機で BT 経路の `register_chipv7_phy` を収束させる**＝**blob のバイトは非収束の «十分条件» ではない＝blob 無罪**。非収束の原因は blob **以外**（グルー／シム／その組合せ）に在る。 | (a) 「**C6 の v5.5.4 統一が可能**」は**言えない**——D-1 は BT の入口であり、D-2a/b/c/d は別途要測定。かつ供給移行には `esp_bt*.cmake` の `ESP_HAL_DIR` 122箇所と `shared_periph_module_t` の解決が別途要る（evidence-c6-01 §9）。**言えるのは「blob はブロッカーではない」まで**。(b) 「hal グルーのどのコードが原因か」は**特定されない**（グルーは bt.c＋シム＋tree ソース群の束）。 |
| **ハング（synth-lock 再現）** | **C3型グルーでも hal 同一 blob は非収束**＝非収束は **{v5.5.4/hal blob のバイト ＋ v5.5.4 tree の非blobソース}** の集合に帰属し、**グルーの «型»（`esp_os_*` vs C3型）では説明されない**。 | 「**blob «だけ» が原因**」とは**言い切れない**——v5.5.4 tree の非blobソースも v6.1 と違う（§5.2）。ただし §12 が発散点を `register_chipv7_phy`（libphy 内部）に局在させているため、**libphy の版差が最も倹約的な説明**である、とは言える。 |

**★どちらに転んでも、v6.1 アームの «この個体での» 実測が要る**（3.5）。

### 3.5 ★個体差の交絡（先回りして潰す設計）

本 DUT は **台帳未知の新個体**。「hal＝ハング／v6.1＝D-1」は**すべて board C**（別個体）の記録であり、
**そのまま移送できない**（本プロジェクトには「board B＝個体難聴＋個体ハングで引退」＝
個体差が実測を誤らせた実績がある）。∴**本ラウンドは同一個体で3アーム全部を回す**：

| # | アーム | 何のために測るか |
|---|---|---|
| 1 | **v5.5.4**（4アーム目） | **本題**＝帰属の決定 |
| 2 | **v6.1**（既定） | **「この個体が壊れている」の排除**。4アーム目がハングしても v6.1 が通れば「個体」ではないと言える |
| 3 | **hal** | **§10-12 のハング baseline を «この個体で» 再現できるか**。★2026-07-15 以降 hal 経路は
`esp_shim_bt_pmu_init` 未定義でリンク不能だった（evidence-c6-01 §5.1）＝**今回リンク可能になって初めて再測できる** |

**★アーム3が «動作» したら**：§10-12 のハングは**この個体では再現しない**＝board C 固有（個体差）の
可能性が立ち上がり、**blob 説もグルー説も «C6一般» としては成立しなくなる**。この分岐を先に書いておく。

### 3.7 計装（`TOPPERS_C6_BT_D1_TRACE`・**既定 OFF**・全アームに同一適用）

C6 実施03 の事故（`no time event is processed in hrt interrupt.` の氾濫が **D-1 到達ログをかき消して
「BLOCKED」と誤診**＝実は達成済みだった）を避けるため、**LP_AON への無条件ミラーを主判定にする**。

`LP_AON_STORE0`（`0x600B1000`・bt_smoke_c6 は未使用／`0x600B101C`＝intr trace のみ使用）へ
段階マーカ `0xB1D000NN` を書く：

| 値 | 意味 |
|---|---|
| `0xB1D00001` | `main_task` 到達 |
| `0xB1D00002` | `esp_shim_bt_clock_init()` 通過 |
| `0xB1D00003` | `esp_bt_controller_init()` 呼出し直前 |
| `0xB1D00004` | `esp_bt_controller_init` OK |
| `0xB1D00005` | **`esp_bt_controller_enable()` 呼出し直前**（★synth-lock ハングはこの中） |
| `0xB1D00006` | **`esp_bt_controller_enable` OK**（＝`register_chipv7_phy` 収束の物証） |
| `0xB1D00007` | HCI Reset 送信 |
| `0xB1D00008` | **D-1 達成**（`hci_reset_done`＝`data[0]==0x04 && data[1]==0x0E` でしか立たない） |
| `0xB1D0000F` | FAILED（Command Complete 来ず） |

**判定規則（先に固定）**：`0x05` で停止＝**synth-lock ハング再現**。`0x06` 以上＝**PHY 収束**。
`0x08`＝**D-1 達成**。

**★マーカは reset を跨いで «残る»**＝過去のブートの値を現在の証拠に誤用しない
（memory: C3/C6 の教訓「adv-rc マーカは reset-survive で «現在放射» を証明しない」）。
⇒ **cleared-boot-read を型として使う**：0クリア→**読んで0を検証**→**1回だけ**ブート→読む。

## 4. 測定計画（実機）

1. `read-mac` で `14:c1:9f:e0:5a:9c` を照合（不一致なら中断）。
2. **真cold**＝`sudo uhubctl -l 1-6 -p 2 -a off` → **`ls /dev/serial/by-id/ | grep -Ec "14:C1:9F:E0:5A:9C|125a266b"` が 0 の読み戻し**（★`rst:` 行は根拠にしない）→ `-a on`。
   **`-l 1-6 -p 2` 以外のポートには触らない**（port1=C3／port3-4=C5／hub 1-5=別プロジェクト）。
3. 採取は**測定対象の状態が始まる前に1回だけ open して保持**（C5 §13 の「capture-open が DUT を
   リセットして «存在しないバグを捏造» した」の再発防止）。
4. 各アーム **独立した真cold 2ブート以上**（rigor 標準＝プラットフォーム主張には ≥2）。
5. `grep -a` 必須。
