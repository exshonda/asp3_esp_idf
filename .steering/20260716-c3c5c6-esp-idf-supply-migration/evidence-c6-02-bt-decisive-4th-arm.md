### 3.7 計装（`TOPPERS_C6_BT_D1_TRACE`・**既定 OFF**・全アームに同一適用）

C6 実施03 の事故（`no time event is processed in hrt interrupt.` の氾濫が **D-1 到達ログをかき消して
「BLOCKED」と誤診**＝実は達成済みだった）を避けるため、**LP_AON への無条件ミラーを主判定にする**。

`LP_AON_STORE0`（`0x600B1000`・bt_smoke_c6 は未使用／`0x600B101C`＝intr trace のみ使用）へ
段階マーカ `0xB1D000NN` を書く：

| 値 | 意味 |
|---|---|
| `0xB1D00001` | `main_task` 到達 |
| `0xB1D00002` | `esp_shim_bt_clock_init()` 通過 |
| `0xB1D00003` | `esp_bt_controller_init()` 呼出し直前 |
| `0xB1D00004` | `esp_bt_controller_init` OK |
| `0xB1D00005` | **`esp_bt_controller_enable()` 呼出し直前**（★synth-lock ハングはこの中） |
| `0xB1D00006` | **`esp_bt_controller_enable` OK**（＝`register_chipv7_phy` 収束の物証） |
| `0xB1D00007` | HCI Reset 送信 |
| `0xB1D00008` | **D-1 達成**（`hci_reset_done`＝`data[0]==0x04 && data[1]==0x0E` でしか立たない） |
| `0xB1D0000F` | FAILED（Command Complete 来ず） |

**判定規則（先に固定）**：`0x05` で停止＝**synth-lock ハング再現**。`0x06` 以上＝**PHY 収束**。
`0x08`＝**D-1 達成**。

**★マーカは reset を跨いで «残る»**＝過去のブートの値を現在の証拠に誤用しない
（memory: C3/C6 の教訓「adv-rc マーカは reset-survive で «現在放射» を証明しない」）。
⇒ **cleared-boot-read を型として使う**：0クリア→**読んで0を検証**→**1回だけ**ブート→読む。

### 5.2 ★予測の答合せ（§3.1）— **両方の予測が «測れていない»**

### 5.3 ★私が途中で犯した誤り（正直に記録する）

## 6. 結論（依頼 (a)〜(g)）

### 6.1 帰属（依頼 (a)(c)）

- **★帰属は決まらなかった＝実験 BLOCKED**。4アーム目は真cold 3/3 でハングしたが、
  **同一個体で v6.1 対照も、board C の成功バイナリそのものもハングする**ため、
  「blob 起因」も「グルー起因」も**言えない**。
- **予測の答合せ＝前ラウンド（ハング）・本ラウンド（動作）とも «未決»**（§5.2）。
  字面上は前ラウンドと同じ結果が出たが、**対照が落ちている以上「的中」と記録してはならない**。

### 6.2 「C6 BT 据置き」判断（依頼 (e)）— **維持する**

- **維持**。ただし**根拠は «blob が黒» ではない**（それは未証明のまま）。
  維持の根拠は「**本ラウンドで判断を動かす新情報がゼロだった**」＝
  実機エビデンス（§13-15）が在るのは依然 v6.1 のみで、それを覆す測定は得られていない。
- **★evidence-c6-01 §6/§7.1 の «blob 起因» 寄りの書き振りは «未証明» に格下げすべき**
  （本ファイルが正本）。`esp_bt_idf61.cmake` の「ハング→blob 起因／動作→グルー起因」という
  **2択の断定コメントは、対照が落ちる個体では成立しない**ことが実測で判明した。

### 6.3 残ブロッカー（依頼 (g)）

1. **★最大＝この DUT では C6 の RF が一切立ち上がらない（BT も WiFi も）**。
   ∴ **本個体では 4アーム目の判定も、WiFi 移行の実機検証も、できない**。
   → **ユーザー判断が要る**：(a) board C（rev v0.3 の個体）を接続する、
   (b) 別の C6（rev v0.3 系）を用意する、(c) 本個体の rev v0.2 対応を調査課題として立てる。
   **これは «判断が要る壁» なので深追いせず止めた**（親指示）。
2. **evidence-c6-01 の WiFi 移行（hal 参照 0）は依然として実機未検証**。
   本ラウンドで初めて実機に載せたが、**個体側の RF が立たないため «移行の可否» を判定できない**
   （hal fallback も同じく落ちるので、**移行の回帰とは切り分け済み＝移行を疑う根拠は無い**）。
3. **§20 pmu_init の «cold PLL-lock を直す» 主張は未検証のまま**（本ラウンドは «無罪» を示しただけで
   «有効» は示していない）。トグル `ESP32C6_BT_PMU_INIT` で今後 A/B 可能。
4. 4アーム目の帰属（blob vs グルー）は**未決のまま次段へ**。**必要なのは
   «v6.1 が D-1 に到達する個体» 1台だけ**——そこで `-DASP3_BT_IDF_V554=ON` を1回焼けば決まる
   （ビルド・計装・手順は本ラウンドで全て用意済み＝`build/c6u_v554`）。

### 6.4 本個体の最終状態

`build/c6_wifiscan_hal`（hal fallback wifi_scan）を書込み済み。**BT 関連の «成功» 状態は無い**
（本個体では存在しない）。次に使う人は焼き直して差し支えない。

### 6.5 ★方法論の収穫（再利用可能）

1. **★LP_AON センチネル＋物理電源断で «真cold» と «マーカの非stale性» を同時に証明できる**：
   `write-mem 0xCAFE5A9C → uhubctl off（by-id 読み戻し 0）→ on → read-mem` が **`0x00000000`**
   ＝(i) 電源断は本物（EN リセットではない）、(ii) **LP_AON マーカは電源断を跨いで残れない
   ＝stale マーカの罠が «構造的に» 消える**。
   ⇒ **「マーカを 0 クリアして read で検証する」手順すら不要になる**（物理が保証する）。
2. **★採取を «一度も開かずに» 真cold boot#1 を観測する型**：電源ON→**18秒放置（capture を開かない）**
   →`esptool read-mem` でマーカ回収。**capture-open が DUT をリセットする問題
   （C5 §13 の «存在しないバグを捏造» 事故）を «原理的に» 回避**できる。
3. **★`rst:0x1 (POWERON)` は今回も証明力ゼロだった**：本DUT の CP2102N は open だけで
   `rst:0x1` を出す（§4 の UART0 配線確認で実証）。**真cold の証明は by-id 読み戻しのみ**。
4. **★C6 の UART0 は «この個体では» 配線されている**（`target.cmake:193` の
   「UART0未配線（Phase Aで確認済み）」は **board C の話であり、個体ごとに違う**）。
   `-DESP32C6_CONSOLE=uart0` が使える＝ROM バナーも app syslog も UART0 に出る。
5. **★「氾濫が出ている＝生きている」のような判別指標は、使う前に «何が出しているか» を確かめる**
   （§5.3 で私が実際に踏んだ）。
