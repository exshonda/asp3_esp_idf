# 実施68〜77の結果

- **実施68**：AGC較正結果値（`ram_pbus_force_mode`/`set_rx_gain_cal_dc_new`）を現行ビルドで再検証→完全一致確認（解決済み）。
- **実施69**：promiscuousモードでもRX無音→フィルタ/ポリシー層は無罪確定。
- **実施70-71**：一過性PHY操作のうち`fe_txrx_reset`だけが差別化要因と判明→`esp_phy_enable()`の**再入**（`s_is_phy_calibrated`既trueでのwakeup高速経路）でのみ発火すると機構を完全特定。nativeは1scan中4回，ASP3は生涯で1回のみ再入。
- **実施72（本調査初の因果介入テスト）**：再入を制御する呼出し連鎖を14段階で追跡し実バグ発見——NuttX互換シムがESP-IDFマクロ名を`CONFIG_ESPRESSIF_WIFI_STA_DISCONNECT_PM`へリダイレクトしており，ASP3の`hal_stub/config.h`がこれを0にハードコードしていた。1へ修正・実機検証：再入サイクルは完全復活（1回→9回）したが，**deaf-RX自体は無変化**——このブランチは反証。
- **実施73→74（重大な自己訂正）**：AGC領域の「決定的リード」（`wifi_agc_sat_gain`でnative=0x828 vs ASP3=0x81825，約254倍）を発見したが，**native参照SDKが全く別バージョンのWiFi PHYブロブ（libphy.a，MD5不一致）をリンクしていたことが判明**——ASP3と正しくblob一致するNuttXで再検証したところ両者とも0x81825で一致，リードは撤回。**以降のバイナリレベル比較は必ずNuttX（blobバージョン一致確認済み）を使う，nativeは定性確認のみという標準ルールを確立**。
- **実施75**：bit14発火瞬間のAGC/MAC全域比較をNuttX対照で正しくやり直し——**AGC領域`0x600a7000`-`0x71ff`は差分ゼロ**（23回連続のbit14発火全てで一致，2ブートずつ）。MAC領域の36差分は全て「受信の結果であって原因ではない」既知パターンに合致。**静的/準静的MMIOスナップショット比較は出尽くした**。
- **実施76**：2台目のC6実機（ボードC）を導入し，**reflash不要の恒久的2台構成**（ボードB=ASP3，ボードC=NuttX，常時同時稼働）を確立。ボードCのnative定性チェック（12 AP検出）でボード健全性を確認後，NuttXを焼いて対照参照として固定。phy_init呼出し列比較を最新ビルドで再実行——0/186 IDミスマッチ，7/186は説明可能な値差のみ。
- **実施77**：regi2cアナログ較正バスの定量的パスを初めて実施。ASP3とNuttXのregi2c操作回数を素朴に比較すると1309 vs 648（約2倍）という「決定的に見えるリード」を発見したが，**同一マイルストーン（`wifi_tr_pos`が186に達した瞬間）へ揃えて再比較したところ反証**——ASP3側の揃えた回数=602（2ブートでバイト単位一致，完全に決定論的），NuttX側は648→692と2ブートで変動（1回目のキャプチャが末尾を静かに切り詰めていたと判明，dumpの`total=`ヘッダ自体から確認）。一見「ASP3だけがブロック0x6a(I2C_BIAS)にアクセスしている」ように見えた差も，NuttX側1回目キャプチャの切り詰めアーチファクトに過ぎないと2回目のブートで判明（自己反証が正しく機能）。唯一残る差：I2C_SAR_ADC reg=6のポーリング回数（ASP3=3 vs NuttX=74-76）だが，**返される値自体は両プラットフォームで完全に同一・安定**（47/0x2F，全読み出しで一致）——アナログ値の乖離ではなく実装上のポーリング回数の違いに過ぎない。

# 現在の到達点：3つの主要な「値レベル」比較軸が全て出尽くした

77ラウンドを通じて，以下は全て反証・解決・除外されました：
- 静的configレジスタ・アンテナ・dwell/timing・RFPLLロック・RXクロック・RX-FEアナログ設定・RXバッファ供給系
- AGC較正結果値（実施68）・フィルタ/ポリシー層（実施69）
- 一過性PHY操作の呼出し有無（`fe_txrx_reset`欠如の根本原因を実際に修正済みだが無関係，実施72）
- **AGC MMIO全域のbit14発火瞬間比較**（実施75，差分ゼロ）
- **phy_init呼出し列の順序・引数比較**（実施70-76，一致）
- **regi2cアナログ較正バスの定量比較**（実施77，操作回数・値とも一致，唯一の差はポーリング回数のみで値は同一）

つまり，**「何を読んでも・何回読んでも値は一致する」というレベルの比較は，2台の実機を使い正しい参照（NuttX，blobバージョン一致確認済み）で，反証規律を保ちながら，出せる限り出し尽くした**というのが現在地です。にもかかわらず，ASP3は一度もRX成功（`lmacProcessRxSucData`，MACイベントbit14）に到達しません。

# 質問

1. **「静的な値・カウント・順序IDは全て一致するのに，一方は受信でき一方は受信できない」という状況で，(a)のタイミング/順序比較（同じ操作が同じ順序で起きるが，タイミング・間隔が違う）は，本当に有望な次の一手でしょうか。それとも，77ラウンドかけて「値」の一致を確認し尽くした今，タイミングも一致する可能性が高く（同じ命令列を実行しているなら，よほど割込みの介在がない限りタイミングも似るはず），別の説明空間へ先に進むべきでしょうか。
2. 実機2台を常時同時稼働できる今の環境で，**最も情報量の多い一手**は何だと思いますか。例えば：
   - regi2cバス以外の，まだ比較していないアナログインターフェース（ADCサンプル値そのもの，AGCのゲイン設定を実際のRF入力に対して動的に変化させているか，等）
   - あるいは，「値は同じだが，物理的な信号／エネルギーがアンテナに到達しているか」という，レジスタでは検証できない層（プロービング・オシロスコープ・別の受信機での傍受等）
   - あるいは，ASP3のDirect Boot自体（ESP-IDFの標準ブートローダ/FreeRTOSを経由しない起動）が，NuttXやnativeでは経由する何らかの**ワンタイムの初期化コード**（ヒューズ読み出し・キャリブレーションデータのロード・ハードウェアの`RFPLL cal` `iq cal`等，起動時に一度だけ実行されるがJTAG接続後には再実行されない類のもの）を根本的に見落としている可能性
3. 77ラウンド・2台の実機・多数の反証を経てなお解決しない場合，**これはこのリポジトリの調査手法（JTAG＋レジスタ比較）の限界に達している可能性**はありますか。その場合，次に投資すべきは何でしょうか（例：Espressifへの問い合わせ，別のESP32-C6ボードでの検証，blobの逆コンパイル解析，等）。
