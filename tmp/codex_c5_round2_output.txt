2. **MODEM/PCR クロック・リセットのパルス履歴**
   スナップショット一致で最も見落としやすいです。reset bit、clock enable、module reset は write-only/self-clear/順序依存になりがちで、最終値一致は因果棄却になりません。raw ADC 恒久ゼロとも整合します。

3. **PMU power-up/isolation FSM のハンドオフ状態**
   `PMU_POWER_PD_*` 系は最終レジスタ値ではなく、「どの順で isolation 解除、clock 供給、reset 解除したか」が内部 FSM/ack/latch に残る可能性があります。PMU 初期値を移植済みでも、同じ handshake を再現していなければ残ります。
   根拠箇所: `components/esp_hw_support/port/esp32c5/pmu_init.c:pmu_hp_system_init()`, `pmu_lp_system_init()`、HAL `hal/esp32c5/include/hal/pmu_ll.h`、関連 `MODEM_PWR0(0x600ad000)`。

4. **clock tree / PLL lock / glitchless mux の過渡状態**
   bootloader と app startup は XTAL/APB/PLL/CPU clock を段階的に切り替えます。PLL lock、modem clock source、divider latch は、読めるレジスタが同じでも内部状態が同じとは限りません。
   根拠箇所: `components/bootloader_support/src/bootloader_clock_init.c:bootloader_clock_configure()`, `components/esp_hw_support/port/esp32c5/clk.c:esp_clk_init()`, `rtc_clk.c:rtc_clk_*`、HAL `clk_tree_ll.h`, `pcr_ll.h`。

5. **ROM/eFuse 由来の一回性アナログ転写**
   regi2c 全域一致と ocode 強制で優先度は下がりますが、ROM/bootloader が eFuse 由来値を write-only analog latch へ流す処理はまだ候補です。
   根拠箇所: `components/esp_rom/esp32c5/ld/*.ld` の `esp_rom_regi2c_*`, `esp_rom_efuse_*`、`components/bootloader_support/src/bootloader_init.c:bootloader_init()`。

6. **PMP/cache/MMU/CPU startup 状態**
   raw ADC ゼロの直接原因としては低めですが、handoff は stock の CPU/system 状態を継承します。Direct Boot と違う `pmpcfg/pmpaddr`, cache/MMU 属性、interrupt/vector 初期化が PHY blob の MMIO 順序や例外処理を変える可能性は残ります。
   根拠箇所: `components/esp_system/port/cpu_start.c:call_start_cpu0()`, `components/esp_system/startup.c:esp_system_init()`。

**二分探索の推奨設計**
最初に切るべきは「boot/startup か、PHY 較正後の残留か」です。

1. **stock minimal app の `app_main` 先頭で即 ASP3 へ handoff**
   NVS、Wi-Fi、`esp_wifi_init` を一切呼ばない。ここで ASP3 側 raw ADC が非ゼロなら、鍵は bootloader または `call_start_cpu0` 以前。ゼロなら、実施26の非ゼロは PHY/Wi-Fi 実行後の残留状態です。

2. **stock 2nd-stage bootloader の通常 app jump 直前で ASP3 へ handoff**
   `components/bootloader/subproject/main/bootloader_start.c` から `bootloader_utility_load_boot_image()` 後、通常の entry jump の代わりに ASP3 へ飛ぶ。ここで成功なら 2nd-stage bootloader/ROM が鍵。失敗で `app_main` 先頭成功なら、鍵は app 側 `call_start_cpu0`/system init。

3. **`call_start_cpu0` 内を分割**
   bootloader 直後失敗、`app_main` 先頭成功の場合に実施。切り方は、`esp_clk_init` 前後、PMU/RTC/periph init 前後、`esp_system_init()` 完了直後、scheduler/app_main 直前。可能なら `ESP_SYSTEM_INIT_FN` の init level で挿す方が、start assembly を触るより安全です。

4. **`app_main` 先頭失敗の場合の PHY 側分割**
   stock app で、`esp_phy_enable` 後、`esp_wifi_init` 後、`esp_wifi_start` 前、scan 後、`esp_wifi_stop/deinit/esp_phy_disable` 後、の順に handoff。どこで ASP3 raw ADC が非ゼロになるかを見る。

実装注意は、handoff 本体を IRAM/DRAM 完結にすることです。MMU unmap/cache invalidate 後に flash 上のコード、rodata、printf/log、libc を踏まない。interrupt/WDT を止め、PMP/memprot は既にやっている通り無効化。bootloader 点では app の heap/BSS/FreeRTOS は存在しないので、app 側 API ではなく bootloader/ROM/MMU HAL だけで完結させるのが安全です。

逆方向は、まず **stock write trace replay** が良いです。ASP3 Direct Boot 後に、stock の POR→app_main 先頭までの PCR/MODEM_SYSCON/MODEM_LPCON/MODEM_PWR0/PMU/LP_CLKRST/PVT への MMIO write を、順序・幅・delay 付きで chunk replay する。regi2c trace は既に一致しているので、次に情報量が高いのは snapshot ではなく transient MMIO write trace です。

**批判と追加実験**
実施26から確実に言えるのは「問題状態は software-reachable で、ASP3 kernel が即座に壊しているわけではない」です。ただし「stock boot が確立した」とまではまだ言い切れません。stock scan/PHY 較正が作った保持状態、または単に done/raw ADC latch が残った可能性が残っています。

安価な識別実験は次です。

- stock `app_main` 先頭即 handoff。これが最重要。
- stock scan 後、handoff 前に `esp_wifi_stop`/`esp_wifi_deinit`/可能なら `esp_phy_disable` してから handoff。
- stock scan 後、handoff 前後で `0x81c..0x828` を時系列読み。ASP3 側で値が時間変化するか、stock 最終値と同一で凍結しているかを見る。
- stock scan 後、modem reset/powerdown を最小限だけ入れてから handoff。成功が消えるなら PHY 残留、残るなら boot/startup 側。
- ASP3 の `_recursive_mutex_create` 実装後、Direct Boot のまま `esp_wifi_init` が raw ADC ゼロを自力で脱出できるか確認。脱出できなければ、OS shim ではなく hidden hardware state 説が強くなります。

次の一手は、`app_main` 先頭即 handoff と bootloader jump 直前 handoff の2点です。この2点だけで、探索範囲は「PHY残留」「2nd-stage bootloader」「app startup」のどれかにかなり鋭く分かれます。

   bootloader 直後失敗、`app_main` 先頭成功の場合に実施。切り方は、`esp_clk_init` 前後、PMU/RTC/periph init 前後、`esp_system_init()` 完了直後、scheduler/app_main 直前。可能なら `ESP_SYSTEM_INIT_FN` の init level で挿す方が、start assembly を触るより安全です。

4. **`app_main` 先頭失敗の場合の PHY 側分割**
   stock app で、`esp_phy_enable` 後、`esp_wifi_init` 後、`esp_wifi_start` 前、scan 後、`esp_wifi_stop/deinit/esp_phy_disable` 後、の順に handoff。どこで ASP3 raw ADC が非ゼロになるかを見る。

実装注意は、handoff 本体を IRAM/DRAM 完結にすることです。MMU unmap/cache invalidate 後に flash 上のコード、rodata、printf/log、libc を踏まない。interrupt/WDT を止め、PMP/memprot は既にやっている通り無効化。bootloader 点では app の heap/BSS/FreeRTOS は存在しないので、app 側 API ではなく bootloader/ROM/MMU HAL だけで完結させるのが安全です。

逆方向は、まず **stock write trace replay** が良いです。ASP3 Direct Boot 後に、stock の POR→app_main 先頭までの PCR/MODEM_SYSCON/MODEM_LPCON/MODEM_PWR0/PMU/LP_CLKRST/PVT への MMIO write を、順序・幅・delay 付きで chunk replay する。regi2c trace は既に一致しているので、次に情報量が高いのは snapshot ではなく transient MMIO write trace です。

**批判と追加実験**
実施26から確実に言えるのは「問題状態は software-reachable で、ASP3 kernel が即座に壊しているわけではない」です。ただし「stock boot が確立した」とまではまだ言い切れません。stock scan/PHY 較正が作った保持状態、または単に done/raw ADC latch が残った可能性が残っています。

安価な識別実験は次です。

- stock `app_main` 先頭即 handoff。これが最重要。
- stock scan 後、handoff 前に `esp_wifi_stop`/`esp_wifi_deinit`/可能なら `esp_phy_disable` してから handoff。
- stock scan 後、handoff 前後で `0x81c..0x828` を時系列読み。ASP3 側で値が時間変化するか、stock 最終値と同一で凍結しているかを見る。
- stock scan 後、modem reset/powerdown を最小限だけ入れてから handoff。成功が消えるなら PHY 残留、残るなら boot/startup 側。
- ASP3 の `_recursive_mutex_create` 実装後、Direct Boot のまま `esp_wifi_init` が raw ADC ゼロを自力で脱出できるか確認。脱出できなければ、OS shim ではなく hidden hardware state 説が強くなります。

次の一手は、`app_main` 先頭即 handoff と bootloader jump 直前 handoff の2点です。この2点だけで、探索範囲は「PHY残留」「2nd-stage bootloader」「app startup」のどれかにかなり鋭く分かれます。
