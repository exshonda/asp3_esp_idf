# C5/C6ブリングアップ調査の知見集約 — S31開発への申し送り

対象読者：ESP32ファミリの新チップ（S31等）へASP3を移植する開発者・調査担当。
出典：`docs/c5-bringup.md` 実施01〜43（C5、真因確定・解決済み）と
`docs/wifi-shim-c6.md` 実施01〜85＋実施86（C6、APM検証）。
本書は「何が起きたか」ではなく「**次の移植で同じ穴に落ちないために何をするか**」に絞る。

## 1. 最重要知見：「Direct Bootのブート列欠落」バグファミリー

C5で発見・修正した8件の真因は、**全て単一のファミリー**に属する：
「ESP-IDFの標準ブート列（2nd-stageブートローダ→`call_start_cpu0`→`esp_system_init`）が
行う初期化を、Direct Bootが実行していない／ROMの既定値を未検証のまま信頼した」。

| # | 真因 | 症状 | なぜ見つけにくかったか |
|---|---|---|---|
| 1 | **APM/TEE未初期化**（WiFi ModemバスマスタがPOR既定のREE2のままHP_APMフィルタに遮断） | TX無・RX無（DMAがパケットバッファに届かない）。それ以外は全部正常 | **CPU（常時TEE）から読めるどのレジスタにも映らない**。バスマスタ視点の遮断は静的比較の原理的盲点 |
| 2 | CPUルートクロックがXTALのまま（PLLへ未切替） | PHY較正のトーン測定が全ゼロ→較正ハング | 「ROMが適切に設定済み」という**前チップ（C6）の実績の無検証流用**。周波数実測で発覚 |
| 3 | ROMのBBPLLがXTAL 40MHzプロファイルで較正（実装は48MHz）→576MHz誤ロック | CPU/ペリフェラルが約20%オーバークロック | 「ロックしている」ことと「正しい周波数でロックしている」ことは別 |
| 4 | CLIC割込み出口（mret非経由経路でmintstatus.mil固着） | 起動72msで全割込み永久凍結 | test_portingが当該経路（wake-from-idle/遅延ディスパッチ）を構造的に通らない |
| 5 | Super WDTアンロックキー誤り（借用値） | scan中に周期リセット | コード自身に「実機未確認」と書いてあった借用値。**キーは絶対に借用しない** |
| 6 | PMU ICGゲート（icg_modem.code=0） | WiFi BBレジスタへの書込みが黙って無効 | 「C6では冗長」の無検証流用（コードに【実機確認待ち】と明記されていた） |
| 7 | regi2cマスタ自体のICGゲート | アナログ設定バスの全書込みが黙って無効 | 書込みが失敗を返さない（黙殺） |
| 8 | ROM残置のPMAエントリ（SRAM実行禁止）、ROMグローバル（`s_ticks_per_us`等）、RTC_FASTソース等 | 個別の機能不全 | stockはstartupで上書きするため、stock環境では絶対に露呈しない |

**S31への適用**：上記は「C5固有のバグリスト」ではなく「**Direct Bootが必ず監査すべき項目リスト**」である。
新チップでは各項目の具体的なレジスタ/値が変わるだけで、穴の構造は同じ。

## 2. アーキテクチャ判断：この問題ファミリーを丸ごと回避する選択肢

**ESP-IDFの`app_main`から`sta_ker`を呼ぶ方式**（asp3_stm32cube/asp3_pico_sdk型）を採れば、
ブート列はstockが全部走るため、ファミリー全体（§1の8件＋将来のIDF版・新チップで増える未知の項目）が
**構造的に発生しなくなる**。C5/C6のハンドオフ実験群（stockブート→ASP3ジャンプでWiFi完全動作）が
この方式の実行可能性を実証済み。

- 留意点：ESP-IDFの`app_main`はFreeRTOS mainタスク上で呼ばれる（stm32cube/pico_sdkのmainと違い
  スケジューラ起動済み）。sta_ker前に「FreeRTOS無力化」（割込み停止・割込みコントローラ奪取・
  タイマ奪取・**ROMインターフェース領域のstale登録ポインタのゼロクリア**——C5実施27）が必要。
  いずれも実証済みの工程。
- **S31の推奨**：新規移植なら**初日からこの方式を第一候補として検討**する。Direct Bootを選ぶ場合は
  §1のリストを移植計画に組み込み、§4の診断チェックリストで検収する。

## 3. 調査方法論の教訓（43＋85ラウンドの授業料）

### 3.1 静的比較には原理的盲点がある——「全レジスタ一致」は「同一状態」を意味しない
C5では、MMIO全域・regi2c全空間・SRAM窓・電源初期化パリティの**全てが一致**した状態で
なおTX/RXが死んでいた。静的スナップショット比較が原理的に見えないもの：
1. **バスマスタ視点のアクセス制御**（APM/TEE）——CPUからの読み書きは常に通るため
2. 書込み専用／セルフクリア型レジスタ、リセット/クロックの**パルス履歴**
3. SRAM上のROM/IDFグローバル（`s_ticks_per_us`型——C6実施48の前例）
4. 過渡シーケンス（PLL再ロック・glitchless mux切替の中間状態）

### 3.2 最初にやるべきだった測定：RX/TXの2×2判定
C5では**実施38になって初めて**「TXが放射しているか」をスニファで測った。結果は「TX無・RX無」で、
これが判明した瞬間に探索空間が正しく絞れた（それまで37ラウンドをRX較正チェーンに費やした）。
**S31では：WiFiが「動くのに通信できない」症状が出たら、最初のラウンドで
(a)スニファでTX放射確認 (b)promiscuousカウンタ (c)MAC割込みレート の2×2判定を行うこと。**
スニファは同世代チップ（デュアルバンドなら両帯監視可能な同型チップ）が最適。

### 3.3 最強のハーネス：クロスカーネル・ハンドオフ
「stockブート→リセット無しMMU再マップ→ASP3ジャンプ」（C6実施33が原型、C5実施26/29で完成）は
本調査の**最重要ツール**だった：
- 「ソフト到達可能か」を一撃で判定（＝ハード故障説・環境説の排除）
- ジャンプ点を動かす二分探索で「鍵の確立段階」を特定（実施29/30）
- 同一バイナリのWORKS/FAILS両系譜を作れる＝**系譜diff**（同一停止点で状態比較）の基盤
- セルフハンドオフ（ASP3→ASP3）で「2回目だから直る」説を判別（実施39）
実装の要点：ハンドオフ本体はIRAM/DRAM完結・stale登録ポインタのゼロクリア・
`CONFIG_ESP_SYSTEM_MEMPROT=n`（実施26。本物ブートローダのPMPロックの回避——C6実施21/22の壁の解）。

### 3.4 例外ラッチレジスタは「物証」になる
APM発見の決め手は**HP_APMの例外ステータス/INFOレジスタに遮断の記録がラッチされていた**こと
（master id・違反アドレスまで読める）。アクセス制御系・エラー系のラッチレジスタは、
静的比較で見えない事象の数少ない「事後観測窓」。**S31の診断では、症状追跡の初期に
APM/TEE・バスエラー・例外系のステータスレジスタを一通り読む**こと。

### 3.5 反証の作法（このリポジトリの厳密性基準の実践形）
- **1実験1機構・事前予測を書いてから実測・予測が外れたら注入に進まず棄却**
- 注入は**読み戻しで着弾を機械確認**（「書いたつもり」の偽陰性が実際に複数回起きた）
- 比較は各側2ブート以上の**4-way**（ブート毎アナログノイズと系譜差の分離）
- 未検証の読取りプロトコルには**known-answer gate**（既知の値を先に読めることを証明してから本番）
- **一次情報は実際にリンクされるツリーで裏取り**（esp-hal-3rdpartyとIDF本体の世代混同で1ラウンド浪費した——実施21）
- 借用値（キー・アドレス・タイミング・「前チップでこうだった」）は全て実機検証対象。
  コードに「実機確認待ち」と書いた仮定は、**書いただけでは守られない**——追跡リスト化して潰す

### 3.6 ツール罠は文書化して引き継ぐ
C5では実機手順の罠が10件超（リセットのドメイン差・JTAG再接続競争・hw watchpoint不発・
esptool自動リセットの較正消費・DTR/ダウンロードモード・tty番号ドリフト等）。
各チップ・各ボードで罠は異なるが、**「罠早見表」をdocsに維持し、各ラウンドの冒頭で必読にする**
運用が有効だった。S31でも初日から同じ表を作ること。

## 4. S31向け：WiFi「静かな失敗」診断チェックリスト（順番どおりに）

1. **APM/TEE**：例外ラッチ読取り→Modem系masterのSEC_MODE→フィルタ状態。
   Direct Bootなら`bootloader_init_mem()`相当（フィルタ無効化 or 全master TEE化）を移植したか
2. **TX/RX 2×2**：スニファでprobe request放射確認＋promiscカウンタ＋MAC割込みレート
3. **CPU実周波数の実測**（mcycle vs 独立タイマの2点法）——「設定したはず」ではなく測る
4. **PLLの較正プロファイル**：実装XTAL周波数とROM/自前較正のプロファイル一致を確認
   （eFuseのXTAL選択と、PLL設定レジスタの分周値の整合）
5. **クロック/電源ゲート**：ICG系（modem・regi2cマスタ）、書込みが黙殺されていないか
   （書いて読み戻す）
6. **WDTキー・PMA/PMP・ROMグローバル**（`s_ticks_per_us`等）・RTCクロックソース
7. 割込み系（CLICチップなら`mintstatus.mil`固着＝§1-4。カーネル時刻が進み続けるか）

## 5. カーネル層の知見（RISC-V/CLICチップ全般）

- **CLIC搭載チップ（C5・C61・H4・P4等）では、TOPPERS RISC-V共通部のmret非経由割込み出口が
  必ずmil固着を起こす**。C5はチップ層のsynthetic mret（実施28）、FMP3はESP32-P4で
  出口正規化により解決済み（`fmp3 doc/clic_design.md`）。両者は双対で不変量は同一
  （精査=`docs/c5-clic-exit-fix-review.md`）。**長期方針はFMP3方式へのasp3_core統一**
  （ユーザ決定済み・C5完了後に実施）。S31がCLICならこの対応を初日から組み込む。
- **test_portingの盲点**：busy-poll構造のため「割込みからのwake-from-idle／遅延ディスパッチ」を
  通らない。当該経路の試験追加をasp3_coreに申し送り済み。新チップの検収でtest_porting 6/6は
  「割込み出口の正しさ」を保証しない点に注意。

## 6. 外部リソースの使い方（本調査で効いたもの）

- **外部AI相談**：行き詰まり時に「確定事実・棄却済み仮説・未検証領域」を自己完結文書化して
  相談する運用（`tmp/external_ai_c5_prompt.md`が雛形）。本調査では候補0（2×2判定）と
  候補3（実行時diff——APM発見の直接の親）が外部AI提案だった。**棄却済みリストを明示する**ことで
  既出提案の往復を防げる。
- **兄弟カーネルの前例確認**：FMP3は同じCLICバグをESP32-P4で先に踏んで解決済みだった
  （設計文書あり）。**新チップで不可解な問題に当たったら、NuttX・FMP3等の同族の
  ポート/設計文書を先に漁る**——1ラウンドで済む調査を自前で数ラウンドかけない。
