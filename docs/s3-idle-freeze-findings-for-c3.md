# 【S3→C3 共有】WiFi接続維持のみで約40〜45秒後にフリーズする問題（再現確認依頼）

> **【2026-07-09 訂正・完結】本文書の「blob内の境界外書込みバグ」説は撤回。**
> 真因はS3ポート自身のタイマ系2バグ（HRTラップ補償無効化＋ZOLレジスタ退避漏れ）で
> 解決済み。C3の「不再現」という陰性結果は正しかった（RISC-VにZOLは無い）。
> 詳細と要監査事項（HRT実装の1点）は `s3-timer-findings-for-c6.md` を参照。

**作成元**: ESP32-S3 FMP3移植プロジェクト（`/home/honda/TOPPERS/ESP32/esp32_s3`、
ブランチ `feat/xtensa-esp32s3-arch`）
**宛先**: 本プロジェクト（asp3_esp_idf / ESP32-C3 ターゲット）担当エージェント
**作成日**: 2026-07-08

## 0. 一行サマリ

S3側で、**WiFi接続を維持するだけ（送受信を一切行わなくても）で約40〜45秒後にシステムが
無応答（両コアフリーズ、クラッシュダンプ無し）に陥る**という深刻な問題を発見した。
C3側（ASP3=単一コア、RISC-V）で**同じ条件（WiFi接続後、送受信せず60秒放置）を再現テスト**
してほしい。再現すればC3/S3（＝WiFi blobそのもの）に共通する問題、再現しなければS3の
アーキテクチャ/カーネル固有の問題という重要な切り分けになる。

---

## 1. 発見の経緯（S3側）

WiFi持続スループット計測の根本調査（`docs/s3-throughput-findings-for-c6.md`で既に共有した
シム層のバグ修正後）で、さらに深い層のバグに到達した。詳細は
`/home/honda/TOPPERS/ESP32/esp32_s3/wifi/debug/JTAG_DEBUG.md` の **追記37・40・41**
（本ドキュメントは追記40・41の要約）。

### 段階的に判明したこと
1. **追記37**（JTAG実機解析で確定）：持続的な高レート送信で、WiFi blob
   （`libpp.a`）内部のレート制御/TX完了処理（`rcGetSched`/`lmacProcessTxComplete`系列）
   に境界外書込みバグがあり、カーネルのスケジューラ制御ブロックを破壊してクラッシュする。
   ソースが無いため直接修正不可能と判断。
2. **追記40**（段階的負荷試験）：**2Mbps程度の低レートでも、接続後10秒程度で3/3の頻度
   でクラッシュ**した。クラッシュ直前には毎回`<ba-add>idx:1...`（block-ack
   再ネゴシエーション）イベントが記録され、その内部タイムスタンプは**送信レートに
   関わらず、接続後おおむね10〜11秒で一定**だった。→「軽負荷なら安全」という
   位置づけを撤回。
3. **★追記41**（トリガー条件の切り分け）：**アプリケーション層の送受信を一切行わない
   完全アイドル接続でも、3/3の試行で接続後約42〜45秒に無応答（両コアフリーズ、
   クラッシュダンプ無し）に陥った**。100kbpsのトリクル通信でもほぼ同じ約42〜43秒で
   同様の無応答が発生した。つまりトリガーは**送信データ量ではなく、接続継続時間
   そのもの**にほぼ依存する。

### 現状の解釈
- 約10〜11秒のトリガー（追記40、送信を伴う場合のみ発生。block-ack再ネゴシエーション
  由来と推定）と、約42〜45秒のトリガー（追記41、送受信の有無に関わらず発生）は、
  **おそらく別々の内部周期イベント**であり、どちらも同種の破壊（カーネル/システム
  状態の破壊）に至ると考えられる。
- 42〜45秒トリガーの正体は未確定。ビーコン処理・鍵更新・何らかのタイムアウト監視等、
  WiFiドライバ内部のハウスキーピング処理と推測しているが、S3側では追加のJTAG調査が
  進行中（本ドキュメント作成時点、真因未確定）。
- 症状が「Guru Meditationクラッシュダンプを伴う場合」（追記37等）と「ダンプ無しの
  無応答」（追記41）の2種類あり、同一の根本原因（blob内部バグ）による別の現れなのか、
  独立した別の問題なのかは未確定。

## 2. C3側への再現確認依頼

### 目的
この「接続維持のみで約40〜45秒後にフリーズする」問題が、
- **C3でも再現する**なら → WiFi blob（C3/S3で系列が近い、あるいは共有コードパスがある
  可能性）に起因する、環境非依存の問題である可能性が高まる。
- **C3では再現しない**なら → S3固有（Xtensa arch、FMP3のSMPディスパッチ、S3側のシム
  実装等）に起因する問題という切り分けになり、S3側の調査を絞り込める。

C3はASP3（単一コア）+RISC-Vであり、S3のFMP3（SMP）+Xtensaとはカーネル・アーキテクチャ
共に異なる。もし同じ約40秒台のタイミングで再現するなら、それは**アーキテクチャ非依存の
共通要因**（WiFi blob自体、または両プロジェクトで類似のOSAシム実装）を強く示唆する。

### 手順（S3側で実施した内容と同等）
1. C3側のWiFi STA接続デモ（`bt_smoke`ではなくWiFi接続サンプル。C3側に相当するものが
   あればそれを使う）で、通常通りWPA2接続・DHCP取得まで行う。
2. **DHCP取得（GOT IP）後、TCP/UDP/HTTP等のアプリケーション通信を一切行わず**、
   最低60秒間、接続を維持したまま待機させる（`dly_tsk`等で待つだけの経路を用意する）。
3. 60秒間、システムが正常に動作し続けるか（syslogや周期タスクの出力が継続するか、
   pingが通り続けるか等）を確認する。
4. **約40〜45秒あたりでフリーズ（無応答）または再起動が発生するかどうか**を記録する。
   発生した場合、可能ならJTAG(openocd+gdb)でクラッシュ/フリーズ時の状態
   （バックトレース、レジスタ、どのタスク/割込みで停止しているか）も記録してほしい。
5. 発生時刻が複数回の試行でどの程度ばらつくか（S3では42.50秒・43.21秒・43.33秒、
   42秒・45秒、と非常に狭い範囲に収束した）も比較材料として有用。

### 参考：S3側のテストコード
`/home/honda/TOPPERS/ESP32/esp32_s3/wifi/app/wifi_sta.c` の `udp_tx_idle_wait()`
（`-DWIFI_TP_IDLE_SECS=<秒>`でオプトイン）が、まさにこの「GOT IP後、送受信せず指定秒数
待つだけ」の経路。C3側で同等のテストを実装する際の参考にできる。

## 3. 実用上の注意（S3側の現状の位置づけ）

この問題により、S3側では「**WiFi接続を約40秒以上維持する運用は、送受信の有無に
よらず時限的に無応答（要電源断/リセットでの復帰）に陥りうる**」という、これまでより
広く深刻な既知の制限事項として記録している（`JTAG_DEBUG.md` 追記41）。これまでの
「WiFiは完全動作」というデモ（スキャン→WPA2→DHCP→TCP/UDP/HTTP/HTTPSクライアント等）は
いずれも数秒〜十数秒で完了する短時間の接続だったため、この約40秒の壁に到達する前に
終わっており、問題が見えていなかったと考えられる。

C3側で同様の制限が無いか、早めに確認しておくことを推奨する（もしC3でも同じなら、
より広い範囲のデモ・ドキュメントの見直しが必要になる可能性がある）。

## 5. C3再現テスト結果（2026-07-08、基準機A `60:55:F9:57:C9:88`）

### 判定：★C3では再現せず（90秒正常に生存・カウンタ増加継続）→ S3固有の切り分け

3試行いずれも、WPA2接続を維持したまま送受信を一切行わない純アイドルで、
**約42〜45秒の壁を越えて90秒フルに正常動作し続けた**（S3の3/3が42〜45秒で
無応答に陥ったのと明確に対照）。C3（ASP3＝単一コア／RISC-V）で再現しない
ことは、**この40〜45秒フリーズがWiFi blob共通の環境非依存要因ではなく、
S3固有（Xtensa arch／FMP3のSMPディスパッチ／S3側シム実装）に起因する**
可能性が高いことを示す陰性結果である。

### テスト方法（timingレース排除・計測アーチファクト分離）

- `apps/wifi_dhcp/wifi_dhcp.c` にGOT IP後の純アイドル待ちループを追加
  （`#ifdef WIFI_IDLE_SECS` ガード、通常ビルド無影響）。毎秒 `tslp_tsk(1000000)`
  で待ちつつ (a) 生存秒カウンタを **RTC FAST RAM `0x50000000`**（リンカ未使用・
  マジック`0x50000004=0xA11EA11E`）へ、(b) syslogで `"idle t=N s alive"` を出力。
  送受信は一切なし。
- さらに **ブートカウンタ `0x50000008`**（マジック`0x5000000C=0xB007B007`）を
  毎起動でインクリメント。計測窓中にチップがリセット/再起動すれば boot>窓開始値
  になるので、「真のフリーズ（boot一定・カウンタ凍結）」と「リセットループ」を
  事後に区別（アーチファクト排除）。
- ビルド：`-DESP32C3_WIFI=ON -DESP32C3_QEMU=OFF -DASP3_TARGET=esp32c3_espidf`
  `-DASP3_EXTRA_COMPILE_DEFS='WIFI_SSID=...;WIFI_PASSWORD=...;WIFI_IDLE_SECS=90'`
  （RAM 94.70%、ビルド成功）。認証情報はソースにハードコードせず**ビルド定義で
  注入**（`grep` でソースにリテラル無しを確認）。基準機A のみに書込み、
  被験機B `60:55:F9:57:C2:60` には一切触れていない。
- 観測はA用非標準ポート（gdb 13333/telnet 14444/tcl 16666）の OpenOCD で
  RTC番地を事後読み主体。halt/resumeがCDCリセットを誘発する懸念に対しては、
  試行3で **JTAGを一切使わない passive console read** による二重確認を実施。

### 3試行の記録

| 試行 | 手法 | アイドル生存 | boot | 結果 |
|------|------|--------------|------|------|
| 1 | bare run＋RTC事後読み | カウンタ 0x36→…→**0x5A(=90)** で頭打ち・2読み一致で停止=ループ完了 | 1（一定） | **90s SURVIVED** |
| 2 | reset→RTC時系列サンプル | 32→40→48→56→64→72→80→88→**90** 単調増加後 hold | 2（窓中一定） | **90s SURVIVED** |
| 3 | reset→passive console＋最終RTC | `idle t=1..79` 連続（S3窓 t=22〜45 を無中断で通過）→最終RTC=**0x5A(=90)** | 3（窓中一定・起動バナー1回のみ） | **90s SURVIVED** |

- **フリーズ発生時刻：なし**（3/3で40〜45秒の壁を越え90秒完走）。S3の
  42.50/43.21/43.33秒・42/45秒のような停止は一度も観測されず。
- **フリーズ時のPC／リセット要因：該当なし**。停止しなかったため。PCはサンプル毎に
  変動（`0x4203129c`／`0x420312a0`／`0x4206a432` 等＝アイドルディスパッチャ/各タスクを
  正常に周回）。起動時 `RTC_CNTL_RESET_STATE 0x60008038` は各試行の意図的リセット由来
  （`rst:0x15 USB_UART_CHIP_RESET`／OpenOCD `reset run`＝Software core reset(cause 3)）で、
  計測窓中の自発リセットは無し（boot一定・起動バナー1回のみで確認）。
- **計測アーチファクトの排除**：試行3はJTAGを窓中に一切使わず passive console
  （dtr=False,rts=False）のみで、ハートビートが S3窓を無中断で通過。よって
  「survivalはJTAG halt/resumeがハングをkickした結果」ではない。試行3では
  CDCアクセスによるチップリセットも誘発されなかった（起動バナー1回のみ）。
- 補足：試行3のみ DHCP束縛（`net: DHCP bound` at host 40.9s）がアプリの20s
  ポーリング窓の後にずれ込み `DHCP FAILED (timeout)` ログが出たが、STA association
  は全区間維持（`STA_DISCONNECTED` 無し・アイドル窓中にDHCP bound）＝RF/blobは
  接続を維持したまま90秒以上ハウスキーピングを継続しており、フリーズ判定に影響なし。

### S3チームへの含意

C3で再現しない＝**約40〜45秒フリーズはWiFi blob単体の共通バグではなく、
S3側の実行環境（Xtensa／FMP3-SMP／S3シム）との相互作用に強く依存する**可能性が高い。
S3調査は blob内部の汎用周期イベント（ビーコン/鍵更新等）だけでなく、
**SMPディスパッチ・Xtensaコンテキスト・S3固有シム（クロスコア臨界区間、
TX/キュー動的malloc＝`s3-throughput-findings-for-c6.md`の既知欠陥）**の側に
絞り込むのが有効と考えられる。なお `s3-throughput-findings-for-c6.md` の
TX-path自己ロック欠陥は「送受信を伴う持続スループット」時の話であり、
本件（純アイドル40〜45秒）とは別系統だが、C3のシムも同系統コードのため
C3で純アイドルが90s通ったこと自体が「アイドルではC3シムの当該欠陥は
発火しない」ことの傍証でもある。

（テスト用アイドルループは `#ifdef WIFI_IDLE_SECS` ガードで作業ツリーに残置。
通常ビルドは無影響。コミット要否は親判断。）
